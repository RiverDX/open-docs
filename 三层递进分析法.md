# 三层递进分析法（3-Layer Progressive Analysis, 3LPA）

> **核心信念**：任何可解决的问题，都可以用 3 个递进的层次来拆解。第一层看清问题，第二层想透问题，第三层动手解决。
>
> **经验法则**：
> - 三层走完 → 解决 90% 的问题
> - 卡在哪一层，对那一层再切 3 刀 → 累计解决 99%
> - 剥到不能再剥时，剩下的不是答案，是问题本身（停手 / 重新提问）

---

## 1. 框架总览

```
                ┌─────────────────────────────┐
                │  第一层 · 定义层（看清）     │  → 目标/标准/范围
                └──────────────┬──────────────┘
                               ↓
                ┌─────────────────────────────┐
                │  第二层 · 结构层（想透）     │  → 拆解/路径/资源/风险
                └──────────────┬──────────────┘
                               ↓
                ┌─────────────────────────────┐
                │  第三层 · 执行层（动手）     │  → 动作/反馈/检查点
                └─────────────────────────────┘
```

每一层只回答一类问题。**层与层之间不混淆**——不要在"还没看清"时跳去"想办法"。

---

## 2. 三层详解

### 2.1 第一层 · 定义层（看清问题）

**回答的是**：这是什么？为什么要做？做到什么样算完？

- **是什么**——用一句话把问题陈述清楚（不要混入解决方案）
- **为什么**——不解决会怎样？解决了价值多大？用来校准投入
- **完成标志**——可观察、可验证的"成了"的样子
- **范围**——什么**不**做（反例往往比正例更省力）
- **时间盒**——最多花多久（避免无限精化）

**关键纪律**：
- 禁止在没写清"完成标志"前进入第二层
- 如果一句话写不下来，说明问题还没被看见，回到原点

**工具集**：
- 5W1H（What/Why/Who/When/Where/How）
- 反例法（什么情况"不算"这个问题）
- 一句话定义（电梯测试：60 秒内能让陌生人听懂）

---

### 2.2 第二层 · 结构层（想透问题）

**回答的是**：怎么拆？怎么走？依赖什么？会卡在哪？

- **关键变量**——列出 5~10 个影响因素，标出"可控 vs 不可控"
- **路径选项**——至少 2 条可行路径 + 各自的代价/收益/风险
- **关键依赖**——人、资源、信息、工具，谁/什么必须到位
- **关键风险**——最可能挂掉的 1~2 个点
- **未知项**——明确"我不知道什么"，优先变成"我要去查什么"

**关键纪律**：
- 用 MECE（不重不漏）拆解，不要堆列表
- 标出**最不可控 + 影响最大**的变量，那是真瓶颈
- 不要在这一层动手做任何执行动作

**工具集**：
- MECE 拆解 / 树状分解
- 关键路径法（CPM）
- 假设-证据配对（每个假设后面挂"我用什么验证"）
- 风险矩阵（影响 × 概率）

---

### 2.3 第三层 · 执行层（动手解决）

**回答的是**：第一步做什么？怎么知道自己在前进？什么时候该升级？

- **24h 第一步**——具体到打开哪个文档、改哪行代码、问谁一句话
- **反馈信号**——短周期可观测的"我在变对"信号（不是最终结果）
- **检查点（Checkpoint）**——每 N 小时/N 步强制回看：是继续？是回第二层？是回第一层？
- **升级条件**——什么情况说明当前路径错了，要重拆
- **止损线**——什么情况说明这个问题不值得再投入

**关键纪律**：
- 第一步必须**24 小时内可完成**（否则就是"宏大叙事"不是"动作"）
- 必须有**短周期反馈**（小时级或天级），否则就是黑盒
- 出了问题**先升级到正确的层**——执行失败通常是第二层（路径）或第一层（定义）错了，不一定是动作本身

**工具集**：
- 最小可行动作（MVA, Minimum Viable Action）
- PDCA / OODA 循环
- 风险预案（Plan B / Plan C 提前想好）

---

## 3. 升级规则（卡壳时怎么走）

| 卡在 | 信号 | 怎么升级 |
| --- | --- | --- |
| 第一层 | 一句话写不出来 / 不知道"什么算完成" | 别动手，再做 3 次"反例 + 完成标志" |
| 第二层 | 拆不出 MECE / 不知道瓶颈 | 把当前最大的子节点单独拉出来，再过 3 层 |
| 第三层 | 做着做着没反馈 / 越做越乱 | 立刻停手，回到第二层复盘路径；不对劲再回第一层 |

**经验法则**：**90% 的卡壳是第二层没想透**。少动手，多想。

## 5. 三个跨领域示例

### 示例 A · 读论文（来自 Keshav）

| 层 | 对应 Keshav 的步骤 |
| --- | --- |
| 第一层（定义） | 第一遍：标题/摘要/引言/结论 + 五个 C |
| 第二层（结构） | 第二遍：忽略证明，抓住主线 + 记未读参考文献 |
| 第三层（执行） | 第三遍：虚拟重做，质疑每个假设 |

### 示例 B · 排查一个生产 Bug

- **第一层**：
  - 是什么：线上服务偶发 500 错误，影响 ~0.5% 请求
  - 完成标志：错误率回到基线，且定位到根因
  - 不做：用户投诉升级、P99 延迟恶化
  - 时间盒：先花 4 小时定位，不行再升级
- **第二层**：
  - 关键变量：上游依赖 / 代码改动 / 流量模式
  - 路径 A：看监控 + 日志关联
  - 路径 B：本地复现 + 打点
  - 路径 C：直接回滚可疑变更
  - 关键风险：可能是间歇性问题，复现不出来
- **第三层**：
  - 24h 第一步：拉最近 1 小时 error log，按 endpoint 分组
  - 反馈信号：错误率随时间变化的曲线
  - 检查点：2 小时没进展就上路径 C
  - 止损：4 小时仍无根因，回滚变更 + 开事故单

### 示例 C · 进入一个新研究方向

- **第一层**：
  - 是什么：在 X 方向做一份能开题的文献综述
  - 完成标志：写出 3000 字综述 + 5 个待回答的研究问题
  - 不做：动手做实验、写代码
  - 时间盒：2 周
- **第二层**：
  - 关键变量：领域成熟度、关键人物、关键会议、数据集可得性
  - 路径 A：找一篇近期综述 → 反向追
  - 路径 B：抓 3~5 篇近作 → 共同被引 + 共同作者
  - 路径 C：上顶级会议 → 扫近 2 年论文集
  - 关键风险：领域术语不熟、方向太宽收不住
- **第三层**：
  - 24h 第一步：Google Scholar 搜 3 个关键词，下载 5 篇高引论文
  - 反馈信号：每篇第一遍"五个 C"都能填出来
  - 检查点：第 3 天必须画出"关键作者-会议-时间线"图
  - 升级条件：第 5 天如果连共同作者都凑不出 5 个，换关键词

## 6. 何时停下（最后那 1%）

剥到第三层还卡着时，先停手问三个元问题：

1. **这个问题"是"什么**——它可能是一个伪问题、一个定义错位、或者一个不可判定命题
2. **我"为什么"要做这个**——回第一层，回到目标层校验
3. **"换一个问题"是不是更划算**——成本-收益重新校准

如果三个元问题都答了仍卡着，**这就是"问题本身"**——接受它是硬核约束，把它记下来，不强求解决。

## 7. 与其他框架的关系

| 框架 | 关系 |
| --- | --- |
| 黄金圈 Why-How-What | 3LPA 第一层的子集（侧重"为什么"） |
| 丰田 5 Whys | 3LPA 第二层的"瓶颈追问"技术 |
| MECE | 3LPA 第二层的拆解纪律 |
| PDCA / OODA | 3LPA 第三层的执行循环 |
| Bloom 认知层次 | 与 3LPA 第一层到第三层近似同构（识记→理解→应用） |

3LPA 不替代这些，它是**把这些工具串成一套顺序**——先看清，再想透，再动手。

---

## 8. 自检清单

每次用 3LPA 走完一遍，对照检查：

- [ ] 第一层有"完成标志"且可验证
- [ ] 第二层有至少 2 条路径 + 标出了瓶颈
- [ ] 第三层有 24h 内可完成的第一步
- [ ] 有短周期反馈信号（≤ 1 天）
- [ ] 有明确的"回上一层"触发条件

5 项都打勾 → 可以放心执行。
