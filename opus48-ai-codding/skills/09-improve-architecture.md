# Skill 09：/improve-architecture —— 深化模块

> **何时启动**：功能完成后；发现代码难理解；每周一次的架构卫生；想让模块变深。
> **目标**：寻找深度模块化机会，不要重写，要渐进改进。
> **核心原则**：深模块（简单接口，丰富实现）；不要重写；一次一个改进。

| **架构决策记录** | `docs/adr/ADR-{N}-{标题}.md`（如有重要决策） | 决策记录 | **长期** |

## 流程（6 步，禁止跳步）

1. **读代码和上下文**：读 CONTEXT.md 和要改进的代码
2. **识别痛点**：哪里难理解？哪里常出错？哪里改起来麻烦？
3. **寻找深模块机会**：什么可以隐藏在简单接口后面？
4. **评估收益/成本**：每个机会带来什么收益？成本多高？风险多大？
5. **建议并选择**：按优先级排序，让用户选择做哪个（如果有）
6. **实施改进**：用 TDD 方式实施（如果用户选择做）

## 寻找深模块机会（从这些角度找）

### 1. 数据簇

**问题**：一些字段总是一起出现，一起变化。

**示例**：
```javascript
// ❌ 分散
function createOrder(userId, amount, currency, status, createdAt) { ... }
function updateOrder(orderId, amount, currency, status) { ... }

// ✅ 聚合成对象
class Money {
  amount: number;
  currency: string;
  
  static of(amount, currency) { ... }
}

class Order {
  money: Money;
  status: OrderStatus;
  createdAt: Date;
}
```

### 2. 调用序列

**问题**：总是需要按固定顺序调用多个方法。

**示例**：
```javascript
// ❌ 调用方需要知道顺序
const validator = new Validator();
const result = validator.validate(input);
if (result.isValid) {
  const processor = new Processor();
  processor.process(input);
}

// ✅ 封装成一个方法
class OrderService {
  validateAndProcess(input) {
    const result = this.validator.validate(input);
    if (result.isValid) {
      this.processor.process(input);
    }
  }
}
```

### 3. 分支逻辑

**问题**：复杂的 if-else/switch，容易错，不好改。

**示例**：
```javascript
// ❌ 复杂分支
if (type === "A") {
  doX();
} else if (type === "B") {
  doY();
} else {
  doZ();
}

// ✅ 策略/状态模式
interface TypeStrategy {
  execute();
}
class StrategyA implements TypeStrategy { ... }
class StrategyB implements TypeStrategy { ... }

context.getStrategy(type).execute();
```

### 4. 低局部性

**问题**：理解一个概念需要在多个文件间跳来跳去。

**改进**：把相关的代码放在一起。

### 5. 泄露的实现

**问题**：接口暴露了太多内部细节，调用方需要知道内部才能用。

**示例**：
```javascript
// ❌ 浅模块：调用方需要知道内部
class Order {
  getItems(): OrderItem[]; // 暴露内部结构
  getTaxRate(): TaxRate; // 暴露细节
  calculateTotal(): number {
    // 调用方需要自己算
  }
}

// ✅ 深模块：简单接口
class Order {
  getTotal(): Money; // 直接给结果，隐藏计算细节
}
```

## 评估框架：收益/成本/风险

对每个机会，评估：

### 收益（高/中/低）

- **可维护性**：代码会更容易理解吗？
- **可测试性**：会更容易写测试吗？
- **可扩展性**：未来改动会更容易吗？
- **减少错误**：会减少 bugs 吗？

### 成本（高/中/低）

- **改动范围**：需要改多少文件？
- **风险**：会影响现有功能吗？
- **时间**：需要多长时间？

### 风险（高/中/低）

- **影响范围**：影响的代码多吗？
- **测试覆盖**：有足够的测试吗？
- **回滚难度**：出问题了容易回滚吗？

---

## 实施改进（如果用户选择做）

**用 TDD 方式**：
1. 🔴 写测试（描述新的接口）
2. 🟢 实现新接口，保持旧接口兼容（如有必要）
3. 🟠 迁移调用方
4. ✅ 删除旧接口（如果安全）

**规则**：
- 一次只做一个改进
- 保持系统始终可工作
- 每个步骤都有测试
- 小步提交，容易回滚

---

## PASS 条件（自查标准，严格解释）

### IA1 寻找了机会
- ✅ 从 5 个角度都找了，至少 2–3 个机会
- ⚠️ 找了但不够全面
- ❌ 没认真找，说"没有改进空间"

### IA2 评估了收益/成本
- ✅ 每个机会都有明确的收益/成本/风险评估
- ⚠️ 有评估但不够清晰
- ❌ 没评估，直接建议改

### IA3 建议渐进改进
- ✅ 都是小的、安全的改进，没有大重写
- ⚠️ 有一些大的改动但有理由
- ❌ 建议"重构整个系统"

### IA4 有具体建议
- ✅ 每个机会都有具体的改进建议，可操作
- ⚠️ 建议有点模糊
- ❌ 只有问题没有建议

### IA5 按优先级排序
- ✅ 按收益/成本比排序了，高收益低成本优先
- ⚠️ 有排序但不太合理
- ❌ 没有排序

### IA6 用 TDD 实施（如适用）
- ✅ 用红→绿→重构方式实施
- ⚠️ 实施了但测试不够
- ❌ 直接改代码，没有测试

## 禁用模式（出现即扣分）

| 禁用做法 | 应改为 |
|---|---|
| "我们需要重写整个系统" | 渐进改进，一次一个小机会 |
| 建议 20 个改进，说"都要做" | 按优先级排序，建议这次只做 1–2 个 |
| 没有收益/成本评估，直接建议改 | 必须评估收益/成本/风险 |
| 改代码不写测试 | 用 TDD 方式 |

---

## 完成后

问用户：
1. "这些机会中，你想做哪个（如果有）？"
2. 如果用户选了，用 TDD 方式实施
3. 完成后，考虑写 ADR 记录决策

**记住**：架构改进是持续的，不是一次性的。每周做一点，代码库会越来越好。
