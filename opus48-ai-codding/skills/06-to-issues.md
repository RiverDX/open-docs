# Skill 06：/to-issues —— 垂直切片

> **何时启动**：PRD 已写完；需要拆分成可执行的任务；准备进入开发。
> **目标**：把 PRD 拆成小的、独立的、垂直切片的 issue，每个都能端到端验证。
> **核心原则**：永远垂直切片，不要水平切片；曳光弹先行。

---

## 输入清单（缺即停）

| 项 | 必需 | 缺失时动作 |
|---|---|---|
| PRD | 必需 | 先跑 `/to-prd` |
| CONTEXT.md | 强烈建议 | 从 PRD 中提取共享语言 |
| 代码库结构（如有） | 建议 | 用 Read 探索项目结构 |

---

## 输出文件（2 个，缺一不可）

| 文件 | 路径模板 | 包含段落 | 寿命 |
|---|---|---|---|
| **Issue 清单** | `docs/issues/{功能名}-{YYYY-MM-DD}.md` | 垂直切片列表 + 依赖关系 + 执行顺序 | **长期**（开发的基准） |
| **Issue 模板** | 每个 issue（或输出建议内容） | 目标 + 验收标准 + 实现指南 | **长期**（直接用） |

---

## 流程（6 步，禁止跳步）

1. **读 PRD**：理解用户故事和验收标准
2. **识别端到端路径**：用户能做的最小但完整的事是什么？
3. **切垂直切片**：每个切片都是一个完整故事，不是一层
4. **标记 AFK/HITL**：哪些 Agent 能自己做，哪些需要人？
5. **识别依赖关系**：哪些必须先做，哪些可以并行？
6. **自查**：按 PASS 条件 AC1–AC6 检查，输出完整清单

## 垂直切片 vs 水平切片

### ❌ 水平切片（不要用）

| Issue | 问题 |
|---|---|
| "建数据库表" | 不能独立验证，半成品 |
| "写 API 层" | 不能独立验证，半成品 |
| "写前端页面" | 不能独立验证，半成品 |

**问题**：
- 单独不能运行
- 交接复杂
- 集成风险大
- 价值交付晚

### ✅ 垂直切片（永远用这个）

| Issue | 说明 |
|---|---|
| "用户可以创建一个草稿并看到在列表中" | 端到端，从 UI 到 DB |
| "用户可以编辑草稿标题并保存" | 端到端，完整功能 |
| "用户可以看到验证错误提示" | 端到端，异常场景 |

**好处**：
- 可独立验证
- 快速交付价值
- 集成风险小
- 优先级清晰

---

## 曳光弹策略

**第一个切片应该是曳光弹**：

- 最小但端到端
- 不求功能完整，但求跑通整条链路
- 像曳光弹一样：打一发，看看方向对不对
- 快速验证架构、工具链、部署流程

**曳光弹例子**：
- 用户可以点一个按钮，看到"Hello World"（但走完整链路）
- 用户可以创建一个空记录，保存在数据库，显示在列表中

---

## AFK vs HITL 标记

### AFK（Away From Keyboard）：Agent 能自己做

特征：
- 需求明确，验收标准清晰
- 不涉及设计决策
- 有相似代码可参考
- 风险低

例子：
- "给用户列表加排序功能"（已有排序模式可参考）
- "在 API 响应中加创建时间字段"

### HITL（Human In The Loop）：需要人盯着

特征：
- 涉及设计决策
- 没有明确的对错
- 高风险
- 第一次做的新功能

例子：
- "设计订单状态机"
- "优化关键查询性能"
- "重构核心模块"

## PASS 条件（自查标准，严格解释）

### AC1 都是垂直切片
- ✅ 每个 issue 都是端到端的完整故事，没有水平切片
- ⚠️ 有一些小的水平切片但有合理理由
- ❌ 明显的水平切片（"先做数据库层"）

### AC2 可独立验证
- ✅ 每个 issue 都能独立运行、独立测试、独立验收
- ⚠️ 有些依赖但基本独立
- ❌ 必须等其他 issue 完成才能验证

### AC3 大小合适
- ✅ 每个 issue 1–2 天能完成，最多不超过 1 周
- ⚠️ 有些稍大但可接受
- ❌ 有超过 1 周的大 issue 没拆分

### AC4 AFK/HITL 标记正确
- ✅ 每个 issue 都标记了，且标记合理
- ⚠️ 有些标记有争议但已记录理由
- ❌ 没标记或明显错误

### AC5 依赖关系清晰
- ✅ 明确列出了依赖关系，执行顺序合理
- ⚠️ 有但不够清晰
- ❌ 没有考虑依赖

### AC6 曳光弹先行
- ✅ 第一个 issue 是曳光弹（最小但端到端）
- ⚠️ 第一个 issue 不是曳光弹但理由充分
- ❌ 没有考虑曳光弹策略

## 禁用模式（出现即扣分）

| 禁用做法 | 应改为 |
|---|---|
| "先做数据库层，再做 API，再做前端" | 垂直切片，一个完整故事 |
| 一个 issue 包含 10 个功能 | 拆分成小的独立 issue |
| 所有 issue 都标记 HITL | 区分真正需要人的和 AFK 的 |
| 验收标准写"完成功能" | 可验证的具体标准 |
