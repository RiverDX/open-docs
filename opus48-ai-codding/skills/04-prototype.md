# Skill 04：/prototype —— 草稿探索

> **何时启动**：遇到高保真问题；需要验证想法；不确定哪种方案更好；想快速看看效果。
> **目标**：用一次性代码回答问题，不要抛光，不要变成生产代码。
> **核心原则**：原型是手电筒，照亮方向后就放下；保留答案，丢弃脚手架。

---

## 输入清单（缺即停）

| 项 | 必需 | 缺失时动作 |
|---|---|---|
| 要回答的核心问题 | 必需 | 反问"这个原型要回答什么问题？用一句话描述" |
| 对比方案（如有） | 建议 | 没有则做一个基准方案 |
| 相关代码（如有） | 建议 | 用 Read 读取关键文件 |

---

## 输出文件（2 个，缺一不可）

| 文件 | 路径模板 | 包含段落 | 寿命 |
|---|---|---|---|
| **原型代码** | `prototype/{方案}/` 或临时目录 | 可运行的原型代码 | **临时**（得到答案后可删除） |
| **原型报告** | `docs/reports/prototype-{目标}-{YYYY-MM-DD}.md` | 问题 + 方案 + 结果 + 建议 | **短期**（记录答案） |

---

## 流程（6 步，禁止跳步）

1. **明确问题**：一句话说清楚要回答什么，成功标准是什么
2. **选择原型类型**：是逻辑原型还是 UI 原型？
3. **设计极简方案**：刚好能回答问题，不多做
4. **构建原型**：快速写代码，不要抛光，不要测试
5. **运行并总结**：运行原型，记录答案
6. **自查**：按 PASS 条件 AC1–AC6 检查，输出报告

---

## 原型的两个分支

### 分支 A：逻辑原型（回答算法、状态、数据问题）

**特点**：
- 命令行脚本，或简单的函数调用
- 显式输出状态和结果
- 如果有多个方案，可切换对比

**示例结构**：
```javascript
// prototype/logical/方案A.js
function algorithmA(input) {
  // ... 实现
}

// 显式输出
console.log("输入:", input);
console.log("结果:", algorithmA(input));
```

### 分支 B：UI 原型（回答交互、体验、视觉问题）

**特点**：
- 单页面（或几个页面）
- 提供切换按钮，可快速对比不同方案
- 不用连真实后端，用 mock 数据
- 不要抛光样式，能用就行

**示例结构**：
```html
<!-- prototype/ui/index.html -->
<button onclick="show('A')">方案 A</button>
<button onclick="show('B')">方案 B</button>

<div id="variant-a" style="display:none">...</div>
<div id="variant-b" style="display:none">...</div>
```

## PASS 条件（自查标准，严格解释）

### AC1 问题明确
- ✅ 一句话说清楚要回答什么，有明确的成功标准
- ⚠️ 问题有点长，但基本清楚
- ❌ 问题不明确，不知道原型要验证什么

### AC2 极简实现
- ✅ 刚好能回答问题，没有多余功能
- ⚠️ 有些多余但不影响
- ❌ 过度设计，做了很多不需要的功能

### AC3 可运行
- ✅ 原型能直接运行，有清晰的运行说明
- ⚠️ 能运行但有点麻烦
- ❌ 不能运行，或者缺少关键依赖

### AC4 多方案对比（如适用）
- ✅ 如果有多个方案，都实现了并可切换对比
- ⚠️ 只有一个方案，但问题确实只需要一个
- ❌ 有多个方案但只做了一个

### AC5 答案清晰
- ✅ 从原型结果能明确回答问题，没有歧义
- ⚠️ 有答案但不够清晰
- ❌ 原型没给出明确答案

### AC6 没有变成生产代码
- ✅ 明确标记为原型，不会被误用为生产代码
- ⚠️ 标记不够明显
- ❌ 原型代码和生产代码混在一起

## 禁用模式（出现即扣分）

| 禁用做法 | 应改为 |
|---|---|
| 原型写得像生产代码一样完整 | 极简，刚好能回答问题就行 |
| 写测试、做重构、抛光 | 这些是生产代码做的，原型不需要 |
| "这个原型代码写得不错，直接用吧" | 不要！原型是一次性的，答案保留，代码丢弃 |
| 试图用一个原型回答 10 个问题 | 一个原型回答一个问题，最多 2–3 个相关的 |

## 完成后

1. 把答案记录到 CONTEXT.md 或 ADR
2. 用 `/handoff` 回到主会话
3. **考虑删除原型代码**（或者明确标记为「不要用」）
4. 建议下一步：是继续 grill 还是进入 `/to-prd`？
