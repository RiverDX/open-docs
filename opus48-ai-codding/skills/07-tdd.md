# Skill 07：/tdd —— 红→绿→重构循环

> **何时启动**：开始实现 issue；写代码前；保证代码质量。
> **目标**：通过测试驱动开发，一次一个测试→实现循环，保证代码质量。
> **核心原则**：先写失败的测试（红）；写刚好让测试通过的代码（绿）；然后重构；通过公开接口测试，不要测试私有方法。

## 流程（6 步，禁止跳步）

1. **理解验收标准**：明确要做什么，不要急着写代码
2. **红**：写第一个失败的测试（只一个，不要多写）
3. **绿**：写刚好让测试通过的代码（不要过度设计）
4. **重构**：在保持测试通过的前提下，改进代码设计（可选，至少想一下）
5. **下一个测试**：重复红→绿→重构，一次只做一个
6. **自查**：按 PASS 条件 TC1–TC6 检查，输出报告

---

## 红→绿→重构详解

### 🔴 红：写失败的测试

**怎么做**：
- 一次只写一个测试
- 从最简单的情况开始
- 通过公开接口调用，不要测试私有方法
- 测试读起来像规格说明
- 确保测试真的失败（不要"绿"着进来）

**好的测试例子**：
```javascript
// 测试读起来像规格说明
test("用户可以创建草稿", () => {
  // 给定（Arrange）
  const user = createUser();
  
  // 当（Act）
  const draft = user.createDraft("标题", "内容");
  
  // 那么（Assert）
  expect(draft.title).toBe("标题");
  expect(draft.status).toBe("draft");
});
```

### 🟢 绿：写刚好让测试通过的代码

**怎么做**：
- 不要过度设计
- 不要预测未来需求
- 可以写"笨"代码（如硬编码）
- 只要通过当前测试就行

**记住**：我们会在重构阶段改进它。

### 🟠 重构：改进设计，保持绿色

**什么时候重构**：
- 看到重复代码
- 看到复杂的条件逻辑
- 看到可以变深的模块
- 看到糟糕的命名

**重构规则**：
- 不要改变行为
- 所有测试必须保持通过
- 一次只做一个改进
- 重构完再写新测试

## 测试原则（严格遵守）

### 原则 1：测试公开接口，不要测试私有方法

| ❌ 不要 | ✅ 要 |
|---|---|
| `test("内部计算方法", () => {` | `test("用户可以创建订单", () => {` |
| `expect(obj._privateMethod()).toBe(...)` | 通过公开 API 测试 |
| 测试实现细节 | 测试行为 |

**为什么**：
- 私有方法可以随意改，测试不该受影响
- 测试读起来像规格说明
- 重构更安全

### 原则 2：垂直 TDD，不要水平 TDD

| ❌ 水平 TDD（不要） | ✅ 垂直 TDD（要） |
|---|---|
| 先写所有测试 | 一个测试→实现→下一个 |
| 再写所有实现 | 红→绿→重构循环 |
| 一次写很多 | 一次一个 |

### 原则 3：测试行为，不是实现

```javascript
// ✅ 好：测试行为
test("订单创建后状态是待支付", () => {
  const order = createOrder();
  expect(order.status).toBe("pending-payment");
});

// ❌ 不好：测试实现细节
test("订单创建后调用了内部方法", () => {
  const spy = jest.spyOn(order, "_updateStatus");
  createOrder();
  expect(spy).toHaveBeenCalled();
});
```

## PASS 条件（自查标准，严格解释）

### TC1 先写测试
- ✅ 总是先写失败的测试，再写实现
- ⚠️ 偶尔跳过但有理由
- ❌ 先写实现再补测试

### TC2 通过公开接口测试
- ✅ 所有测试都通过公开接口调用，不测试私有方法
- ⚠️ 有少量测试直接调用内部但有理由
- ❌ 大量测试私有方法

### TC3 一次一个循环
- ✅ 严格执行红→绿→重构，一次一个测试
- ⚠️ 偶尔一次写两个但不多
- ❌ 先写所有测试再写所有实现

### TC4 测试读起来像规格
- ✅ 测试用业务语言写，读起来像说明文档
- ⚠️ 有点技术但基本清晰
- ❌ 测试只有代码，没有意义

### TC5 覆盖边界条件
- ✅ 覆盖了正常路径 + 边界条件 + 异常场景
- ⚠️ 有边界条件但不全
- ❌ 只测 happy path

### TC6 没有恒真断言
- ✅ 所有断言都有实际意义，不会永远为真
- ⚠️ 有一些不太好的断言
- ❌ 有 `assertNotNull(result)` 这种没意义的断言

## 禁用模式（出现即扣分）

| 禁用做法 | 应改为 |
|---|---|
| 先写 10 个测试，再写所有实现 | 一次一个测试→实现循环 |
| 测试私有方法 | 通过公开接口测试行为 |
| 为了测试把私有方法改成 public | 重新设计，或通过公开接口测 |
| `assertNotNull(result)` 这种没意义的断言 | 断言具体的行为或值 |
| 重构时改变了行为 | 重构时保持所有测试通过，只改设计不改行为 |
| 过度设计，预测未来需求 | 写刚好满足当前测试的代码 |
