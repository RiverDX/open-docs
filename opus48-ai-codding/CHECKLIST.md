# Opus48 检查清单

> **任何时候都可以用这个清单自查。**

---

## 🎯 总体健康度

- [ ] 我理解了问题，没有急着写代码
- [ ] 我在聪明区工作（< 120K tokens）
- [ ] 我用的是垂直切片，不是水平切片
- [ ] 我通过公开接口测试，不是测试私有方法

---

## 1. /triage —— 画布整理检查清单

- [ ] 每个 issue 都有类别标签（bug / enhancement / question）
- [ ] 每个 issue 都有状态标签（needs-triage / needs-info / ready-for-agent / ready-for-human / wontfix）
- [ ] `ready-for-agent` 的 issue 有清晰的验收标准
- [ ] `wontfix` 的 issue 有明确的原因记录
- [ ] 没有"太大"的 issue，都拆成了可管理的小任务

---

## 2. /grill-with-docs —— 对齐共识检查清单

- [ ] 我理解了用户的真实意图，不是只看字面意思
- [ ] 我问了足够多的问题（至少 5 个实质性问题）
- [ ] 我澄清了所有边界条件和异常场景
- [ ] 我用 CONTEXT.md 记录了共享语言
- [ ] 我识别了需要原型验证的高保真问题
- [ ] 我没有编造行号或代码位置
- [ ] 所有不确定项都打了 `[待验证]` 标记

---

## 3. /handoff —— 上下文接力检查清单

- [ ] 我明确了下一个会话的目标（一句话说清楚）
- [ ] 我只包含了相关的上下文，没有把整个对话复制过去
- [ ] 我引用了文件路径，没有重复文件内容
- [ ] 我记录了已经达成的决策
- [ ] 我列出了下一个会话需要做的明确任务
- [ ] 我建议了下一个该用的 Skill
- [ ] 我脱敏了所有敏感信息（API key、密码等）

---

## 4. /prototype —— 草稿探索检查清单

- [ ] 我明确了要回答的核心问题（一句话说清楚）
- [ ] 我的原型是极简的，刚好能回答问题
- [ ] 我没有过度设计或抛光
- [ ] 如果是 UI 原型，我提供了可切换的变体
- [ ] 如果是逻辑原型，我提供了清晰的输出来比较
- [ ] 我记录了答案，但没有把原型代码变成生产代码
- [ ] 我建议了下一步：是回到 grill，还是进入 to-prd？

---

## 5. /to-prd —— 定稿草图检查清单

- [ ] 我记录了问题陈述（我们解决什么问题）
- [ ] 我记录了解决方案（我们做什么，不说"怎么做"）
- [ ] 我用了用户故事（作为 X，我可以 Y，以便 Z）
- [ ] 每个用户故事都有可验证的验收标准
- [ ] 我识别了深模块机会（简单接口 + 丰富实现）
- [ ] 我列出了主要模块和它们的职责
- [ ] 我明确了范围外事项（我们不做什么）
- [ ] 我没有重新 grill 用户，只用已有共识
- [ ] 不确定项都打了 `[待验证]` 标记

---

## 6. /to-issues —— 垂直切片检查清单

- [ ] 每个 issue 都是垂直切片（端到端的完整故事）
- [ ] 每个 issue 都能独立验证
- [ ] 没有"先做数据库层，再做 API 层"这种水平切片
- [ ] 每个 issue 都标记了 AFK 或 HITL
- [ ] 我识别了依赖关系，有合理的执行顺序
- [ ] 第一个 issue 是曳光弹（最小但完整）
- [ ] 验收标准清晰，没有"大概完成就行"

---

## 7. /tdd —— 红→绿→重构检查清单

- [ ] 我先写了失败的测试（红）
- [ ] 我写了刚好让测试通过的代码（绿）
- [ ] 我没有过度设计，没有预测未来需求
- [ ] 我通过公开接口测试，不是测试私有方法
- [ ] 我的测试读起来像规格说明
- [ ] 我覆盖了边界条件和异常场景
- [ ] 我重构了，但没有改变行为
- [ ] 所有测试仍然通过
- [ ] 我一次只做一个测试→实现循环

---

## 8. /codex-review —— 画布审查检查清单

- [ ] 我检查了是否违反 CONTEXT.md 中的约定
- [ ] 我检查了明显的 bug（空指针、并发、事务、安全）
- [ ] 我检查了测试覆盖度（关键路径都有测试吗？）
- [ ] 我检查了命名和包结构一致性
- [ ] 我检查了是否有浅模块可以变深
- [ ] 我用了 SEV-1 / SEV-2 / SEV-3 分级
- [ ] SEV-1 问题不超过 5 条（避免淹没）
- [ ] 我给出了具体的文件:行号引用
- [ ] 我没有编造问题，只报告真实发现

---

## 9. /improve-architecture —— 深化模块检查清单

- [ ] 我没有提议重写整个系统
- [ ] 我找的是小的、渐进的、安全的改进
- [ ] 每个改进都有清晰的收益（局部性、可测试性、简单接口）
- [ ] 我评估了风险（改动范围、对现有功能的影响）
- [ ] 我按收益/成本排序了机会
- [ ] 我让用户选择做哪个（如果有的话）
- [ ] 我用 TDD 方式实施改进
- [ ] 我更新了 CONTEXT.md 或 ADR（如果需要）

---

## 10. /doc-rot —— 月度巡检检查清单

- [ ] 我检查了 CONTEXT.md 中的引用是否仍然有效
- [ ] 我检查了 docs/entry-points.md 中的接口是否与代码一致
- [ ] 我检查了 docs/glossary.md 中的术语是否仍在使用
- [ ] 我检查了过去一个月合并的 PR 中是否有变更未同步到文档
- [ ] 我用了 P0/P1/P2 优先级
- [ ] P0 问题不超过 5 条（避免淹没）
- [ ] 我计算了总体健康度评分
- [ ] 我建议了下一步：是回触 reverse-loop，还是优化 skill？

---

## 🔴 紧急停止清单

如果出现以下任何一种情况，**立即停止**，不要继续：

- [ ] 我不知道用户要做什么，但我在猜
- [ ] 用户没有回答我的问题，但我在继续推进
- [ ] 我需要的关键代码缺失，但我在编造
- [ ] 当前 tokens 已超过 120K，我还没有提议 handoff
- [ ] 测试没有通过，但我在写更多代码
- [ ] 我在重写整个模块，而不是渐进式改进
- [ ] 我在测试私有方法，而不是通过公开接口
- [ ] 我在水平切片（先做数据库层，再做 API 层）
