## 什么时候用哪个 Skill？

| 场景 | 用哪个 Skill |
|---|---|
| 有一堆 idea/issue 需要整理 | `/triage` |
| 需求不清晰，概念有歧义 | `/grill-with-docs` |
| 对话太长，接近变笨区 | `/handoff` |
| 遇到高保真问题，需要验证想法 | `/prototype` |
| 共识已达成，需要写 PRD | `/to-prd` |
| PRD 已写，需要拆成任务 | `/to-issues` |
| 开始实现，要保证代码质量 | `/tdd` |
| 代码写完，提交 PR 前 | `/codex-review` |
| 功能完成，想改进架构 | `/improve-architecture` |
| 定期检查文档是否过时 | `/doc-rot` |

## 核心概念 10 秒了解

### 🧠 聪明区 vs 变笨区
- **聪明区**：<120K tokens，AI 专注、有创造力
- **变笨区**：>120K tokens，注意力分散、决策质量下降
- **怎么办**：用 `/handoff` 分流

### 📦 深模块 vs 浅模块
- **深模块** ✅：简单接口，丰富实现（调用方不需要知道内部）
- **浅模块** ❌：复杂接口，简单实现（调用方需要知道所有细节）
- **目标**：让模块变深

### 🎯 垂直切片 vs 水平切片
- **垂直切片** ✅：用户可以端到端做一件事（如"用户可以创建草稿并看到在列表中"）
- **水平切片** ❌：只做一层（如"先做数据库层"）
- **永远用垂直切片**

### 🤖 AFK vs HITL
- **AFK**（Away From Keyboard）：Agent 能自己做（需求明确、风险低）
- **HITL**（Human In The Loop）：需要人盯着（设计决策、高风险）

## 新手第一天：试试这 3 个

### 1️⃣ /grill-with-docs
下次有需求时，先别急着写代码，用这个技能提问：
- 我们解决什么问题？
- 这个概念叫什么？确切含义？
- 边界在哪里？什么不做？

把共识记在 CONTEXT.md。

### 2️⃣ /tdd
写代码前，先写测试：
1. 🔴 写一个失败的测试（读起来像规格说明）
2. 🟢 写刚好让测试通过的代码
3. 🟠 重构（可选）
4. 重复

### 3️⃣ /doc-rot
月底时，花 15 分钟：
- 检查 CONTEXT.md 还对吗？
- API 文档和代码一致吗？
- 计算健康度评分

## 老项目接入路径

### 第 1 周：摸底
1. 跑 `/doc-rot` 做一次完整巡检
2. 看看健康度多少，最大的问题在哪里

### 第 2–3 周：整理
1. 对最腐化的模块跑 `/grill-with-docs` + `/improve-architecture`
2. 更新 CONTEXT.md

### 第 4 周开始：新流程
1. 新功能用完整流程：`/grill-with-docs` → `/to-prd` → `/to-issues` → `/tdd` → `/codex-review`
2. 每周做一次 `/improve-architecture`
3. 每月做一次 `/doc-rot`

## 检查清单（随时查）

### 写代码前
- [ ] 我理解需求了吗？（不够的话用 `/grill-with-docs`）
- [ ] 我在聪明区吗？（不在的话用 `/handoff`）
- [ ] 这是垂直切片吗？（不是的话重新切）

### 写代码时
- [ ] 我先写测试了吗？（用 `/tdd`）
- [ ] 我通过公开接口测试吗？（不是的话改）
- [ ] 我一次只写一个测试→实现循环吗？

### 写完代码后
- [ ] 我用 `/codex-review` 预审了吗？
- [ ] 有可以变深的模块吗？（用 `/improve-architecture`）
- [ ] 文档需要更新吗？（用 `/doc-rot` 检查）

## 常见问题

### Q：我不想一下子用所有 Skill，太复杂了
**A**：没问题！从这 3 个开始：
1. `/grill-with-docs`：先理解再动手
2. `/tdd`：测试先行
3. `/doc-rot`：每月检查文档

习惯了再加其他。

### Q：Grill 多久算够？什么时候可以写代码？
**A**：当你觉得：
- 概念清晰，没有歧义
- 边界明确，知道什么做什么不做
- 验收标准可验证

**不要追求完美**，80% 清楚就可以开始，剩下的边做边调整。

### Q：原型代码要保留吗？
**A**：不要！保留答案，丢弃脚手架。
- 原型是手电筒，照亮方向后就放下
- 把学到的记录到 CONTEXT.md 或 ADR
- 原型代码删除或明确标记为「不要用」

### Q：我知道模块可以变深，但现在没时间
**A**：没关系！用 `/improve-architecture` 把机会记录下来，以后再做。
- 每周留 1–2 小时做架构改进
- 不要追求完美，小步前进
