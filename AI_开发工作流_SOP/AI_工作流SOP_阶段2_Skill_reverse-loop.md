# Skill：reverse-loop —— 逆向理解的 Loop 版本

> **定位**：把 SOP 阶段 2 中的 `reverse-mapping`（单次提问版）升级为 **Loop 版本** —— 让 AI 自己按验收标准自查、修正、停止，专为「小白也能跑得动」与「让 AI 输出更可信」设计。  
> **关联文件**：`AI_工作流SOP_阶段2_逆向理解_操作手册.md`、`AI_开发工作流_流程性_SOP.md`  
> **理论基础**：Loop Engineering 的 4 模块（目标 / 步骤 / 检查 / 停止）。  
> **版本**：V1.0（2026-06-23）

---

## 1. 目标

让一段陌生的 Java / SpringBoot 代码，经过一次 AI Loop，被还原成「人看得懂、AI 看得懂、新人接手能改」的模块文档，且**输出本身带自查证据和待人工确认清单**，不需要你再回头检查 AI 是否在编。

与 `reverse-mapping` 的差异：

| 对比项 | reverse-mapping（旧） | reverse-loop（新） |
|---|---|---|
| 交互模式 | 单次提问 → 单次输出 | 多轮自查 → 收敛输出 |
| 验收方式 | 无（靠人工事后检查） | AI 自带打分表 + 证据行号 |
| 越界控制 | 无（可能越改越偏） | 最多 2 轮，超时即输出"卡点报告" |
| 不确定信息 | 容易被当成事实写入 | 强制打上 `[待验证]` 标签 |
| 适合人群 | 已熟悉 prompt 工程的开发者 | 小白也能直接复制运行 |

---

## 2. 触发条件

满足任一条即可启用 reverse-loop 而不是 reverse-mapping：

- 第一次接触某模块，需要给非作者也能看懂的产物；
- 模块代码量 > 500 行，单次 prompt 容易丢细节；
- 要把模块文档交给业务方或新人 review；
- 已经跑过一次 reverse-mapping，但产出有明显"AI 编"的痕迹；
- 准备进入 SOP 阶段 3（共识确认），需要产出可被 review 的稳定文档。

不需要 Loop 的场景（用 reverse-mapping 即可）：

- 只看一个 200 行以内的小工具类；
- 只想快速了解某个方法的大致功能；
- 已经熟悉该模块，只是补一份摘要。

## 3. 输入

| 输入项 | 是否必需 | 说明 |
|---|---|---|
| 模块名 / 边界 | 必需 | 一个明确的模块、功能或链路名 |
| 入口 | 必需 | Controller 路径、消息 topic、定时任务方法名等 |
| 关键代码 | 必需 | Controller、Service、Mapper、Entity 至少各 1 个 |
| 根 `CLAUDE.md` | 强烈建议 | 项目级约束 |
| 模块级 `CLAUDE.md`（如有） | 可选 | 模块级约束 |
| 数据库 schema | 强烈建议 | 表结构 DDL 或字段说明 |
| 现有文档 / API 描述 | 可选 | 作为交叉验证依据 |

---

## 4. 输出

跑完一次 Loop，AI 必须输出 **3 段内容**：

### 4.1 最终模块文档（可直接存为 `docs/modules/{模块名}.md`）

结构同 SOP 阶段 2 操作手册步骤 7，但每条都需引用具体代码行号。

### 4.2 自查表（核心新增产物）

```markdown
| 验收标准 | 评分 | 证据 / 卡点 |
|---|---|---|
| 业务职责一句话+一段话且引用 2 处代码 | ✅ / ⚠️ / ❌ | OrderService.java L42, L88 |
| 每个接口含方法/路径/参数/事务边界 | ✅ / ⚠️ / ❌ | ... |
| Mermaid 流程图每节点对应一个方法 | ✅ / ⚠️ / ❌ | ... |
| 每个实体对应表名，状态字段列出枚举 | ✅ / ⚠️ / ❌ | ... |
| 至少 3 条风险，含文件行号 | ✅ / ⚠️ / ❌ | ... |
| 不确定描述均带 [待验证] | ✅ / ⚠️ / ❌ | ... |
```

### 4.3 需人工确认清单

```markdown
- [ ] 卡点 1：xxx（建议怎么确认）
- [ ] 卡点 2：xxx
- [ ] 待验证项 1：xxx（建议如何验证：跑接口 / 查数据 / 问业务方）
```

## 5. 通用 Prompt（核心模板，小白可直接复制粘贴）

```markdown
# 角色
你是正在接手一个陌生 Java / SpringBoot 模块的资深开发者，目标是用 Loop 方式自我闭环地完成模块逆向理解。

# 任务
还原模块 {模块名} 的业务逻辑与技术结构，让一个从未接触过此代码的人也能看懂。

# 流程（必须按此顺序执行，不要跳步）
1. 扫一遍代码结构，列出 Controller / Service / Mapper / Entity 文件清单
2. 按 "入口 → 核心 Service → 数据访问 → 外部调用" 分 4 批阅读
3. 输出第一版理解（业务职责 / 接口 / 流程 / 数据模型 / 风险）
4. 按下方"验收标准"逐项自查并填入自查表（✅/⚠️/❌ + 证据行号）
5. 对 ⚠️/❌ 项尝试自行修正一轮（最多 2 轮）
6. 输出最终文档 + 自查表 + 需人工确认清单

# 验收标准（每项必须有具体证据，否则记 ⚠️ 或 ❌）
- AC1 业务职责：能用 1 句话 + 1 段话讲清，至少引用 2 处具体代码行
- AC2 接口与入口：每个接口列出 HTTP 方法 / 路径 / 关键参数 / 事务边界
- AC3 核心流程：用 Mermaid 画出，每个节点对应一个具体方法名
- AC4 数据模型：每个实体对应到具体数据库表名，状态字段列出全部枚举值
- AC5 风险与异味：至少 3 条，每条注明具体文件 + 行号
- AC6 不确定项：所有"可能 / 似乎 / 大概"的描述必须加上 [待验证] 标签

# 停止条件
- 全部 AC1–AC6 评为 ✅ → 立即停止，输出最终结果
- 已自查修改 2 轮仍有 ⚠️/❌ → 停止，不要继续猜
  · 在"需人工确认清单"里明确写出：卡在哪里 + 缺什么信息 + 建议如何验证
- 严禁为追求完美而无限改写已经合格的部分

# 输出格式（严格遵循）

## 6. Claude Code 适配

把上面的通用 prompt 保存为 `.claude/commands/reverse-loop.md`（项目内 slash command 形式）：

```markdown
---
description: 用 Loop 方式逆向梳理一个模块的业务与技术结构
argument-hint: <模块名>
---

请用 reverse-loop 模式还原模块 $1 的业务逻辑。

# 必须做的事
1. 先扫 src/main/java/**/$1/ 下的全部文件，列出结构
2. 按 入口 → Service → 数据访问 → 外部调用 顺序读代码
3. 按本 skill 文档（docs/ai-skills/reverse-loop.md）的 6 条验收标准 (AC1-AC6) 自查
4. 最多自查 2 轮，超出即停并输出"卡点报告"

# 输入参考
- 根 CLAUDE.md：见仓库根目录
- 模块级 CLAUDE.md：见 src/main/java/**/$1/CLAUDE.md（如有）
- 数据库 schema：见 docs/data-model.md 或 src/main/resources/db/

# 输出
1. 写入 docs/modules/$1.md（如已存在则附加 "## 第 N 次逆向" 章节）
2. 自查表 + 需人工确认清单写入 docs/modules/$1-loop-report.md
3. 输出到对话末尾的简要摘要：通过的 AC、未通过的 AC、需人工动作
```

使用方式：

```
/reverse-loop order
```

## 8. 注意事项 / 常见陷阱

### 8.1 小白最容易踩的 5 个坑

1. **不填验收标准，直接复制模板**  
   → AC1–AC6 是固定的，但每个项目可以加自定义 AC（如"必须列出所有 @Transactional 方法"）。

2. **代码贴太多，超过 AI 上下文**  
   → 严格按"入口 → Service → 数据访问 → 外部调用"分批，每批 ≤ 500 行。

3. **一上来就要 AI 总结整个微服务**  
   → 一次只逆向一个模块 / 一条链路。微服务先画拓扑图，再分服务逆向。

4. **看见 AI 输出 ✅ 就信了**  
   → 自查表只是 AI 的自我评估，等同于「初稿质检」。人工至少抽查 2 个 AC 的证据是否真的存在于代码中。

5. **AI 输出 ❌ 时强迫它继续**  
   → ❌ 意味着信息不足，逼它只会编造。正确做法是按"需人工确认清单"补料后再跑一次。

### 8.2 AI 端最容易出错的环节

| 出错点 | 表现 | 应对 |
|---|---|---|
| 事务边界 | AI 经常把 `@Transactional` 写在错的层 | 在 prompt 里强制要求列出所有 `@Transactional` 方法 |
| 异步消息 | 忽略 `@Async`、Kafka 监听器、定时任务 | 在 AC2 中明确要求"含 HTTP / 消息 / 定时任务全部入口" |
| 状态枚举 | 只写常见状态，漏掉异常分支 | 在 AC4 中要求"列出 enum 全部值，含已废弃的" |
| 行号引用 | 编造行号 | 人工抽 2 处验证；发现编造立即在自查表里降级 |

### 8.3 与其他 Skill 的衔接

- **跑完 reverse-loop 后**：把"需人工确认清单"转化为 SOP 阶段 3「共识确认」的检查项；
- **如果项目有 `/grill-deep`**：可以把 reverse-loop 输出的 `[待验证]` 项作为 grill-deep 的输入；
- **跑出的 `docs/modules/{模块名}.md`**：直接进 SOP 阶段 3 的 review 流程。

## 9. 一次最小可用示例（订单模块）

**输入**：

```
模块名：order
入口：POST /api/v1/orders
关键代码：OrderController.java (120 行), OrderService.java (380 行),
        OrderMapper.java (60 行), Order.java (80 行), OrderItem.java (50 行)
schema：orders, order_items 两张表的 DDL
```

**调用**：在 Claude Code 里执行 `/reverse-loop order`，或在 Cursor 里粘贴 §5 的通用 prompt。

**期待产出**：

- `docs/modules/order.md`（最终文档）
- `docs/modules/order-loop-report.md`（自查表 + 待确认清单）
- 对话末尾简要摘要，例如：

```
✅ AC1, AC2, AC3, AC5（已通过）
⚠️ AC4：order_items.status 字段在代码中只见到 0/1/2，但 DDL 默认值是 9，未确认是否有"已退款"状态
⚠️ AC6：3 处 [待验证] 标签，主要在跨服务回调幂等性

需人工动作：
1. 找 PO 确认 order_items.status=9 的含义
2. 抓一次回调失败的日志验证幂等机制
3. 跑一次 OrderControllerIntegrationTest 验证事务回滚行为
```

这就是一次合格的 reverse-loop 输出 —— **不是"AI 帮你做完了"，而是"AI 把哪些做完了、哪些没做完、需要你做什么"都讲清楚了**。
