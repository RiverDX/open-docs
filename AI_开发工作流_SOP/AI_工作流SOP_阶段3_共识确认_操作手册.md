# 阶段 3：共识确认 —— 操作手册

> **目标**：把逆向理解的产出转化为经过人工确认、可进入正向迭代的输入。  
> **前提**：阶段 2 已完成，生成了模块文档、术语表、入口地图等。  
> **预计耗时**：单个模块约 1–3 小时；涉及业务方确认时可能延长。

---

## 1. 所需工具

| 工具 | 用途 | 是否必需 |
|---|---|---|
| **AI 工具**（Claude Code / Cursor / Copilot / 通义灵码） | 生成 PRD、拆 Issue、辅助 review | 是 |
| **Markdown 编辑器**（Obsidian / VS Code） | 查看和编辑共识文档 | 是 |
| **沟通工具**（飞书/钉钉/腾讯会议/微信） | 与业务方/技术负责人确认 | 业务复杂时必需 |
| **Issue 跟踪工具**（GitHub Issues / GitLab Issues / Jira / TAPD） | 创建开发任务 | 建议 |
| **Git** | 管理文档版本 | 是 |

---

## 2. 环境准备

### 2.1 收集待 review 材料

确保以下材料已准备就绪：

- `docs/modules/{模块名}.md`
- `docs/glossary.md`
- `docs/entry-points.md`
- 当前根 `CLAUDE.md` 和模块级 `CLAUDE.md`
- 原始需求描述（如果有）
- 相关业务方或技术负责人的联系方式

### 2.2 确定 review 方式

| 方式 | 适用场景 | 耗时 |
|---|---|---|
| 异步文档 review | 变更小、参与人少 | 0.5–2 小时 |
| 线上会议 | 业务复杂、涉及多方 | 1–2 小时 |
| 线下白板/文档 | 首次接手大型模块 | 2–4 小时 |

## 3. 操作步骤

### 步骤 1：自审 AI 产出

在交给他人 review 前，先自己过一遍：

```markdown
# 自审清单

1. AI 描述中是否有“可能”、“似乎”、“也许”等不确定词？
2. 是否有与代码实际不符的地方？
3. 是否遗漏了关键业务规则或边界条件？
4. 状态流转是否完整？
5. 外部接口/消息契约是否准确？
6. 数据一致性要求（事务、幂等、补偿）是否明确？
```

把不确定项标记为 **“待验证”** 或 **“待确认”**，不要直接写死。

---

### 步骤 2：人工 review

#### 2.1 选择 review 人

- 技术侧：熟悉该模块的开发者、架构师；
- 业务侧：产品经理、业务负责人、运营（如果涉及业务规则）；
- 测试侧：测试负责人（帮助确认边界条件和验收标准）。

#### 2.2 review 重点

| 维度 | 审查内容 |
|---|---|
| **业务正确性** | 术语是否准确？流程是否符合实际业务？ |
| **边界条件** | 异常场景、空值、并发、超时、重试是否覆盖？ |
| **数据一致性** | 事务边界、幂等、补偿、缓存一致性是否清晰？ |
| **接口契约** | 入参、出参、状态码、错误码是否与线上一致？ |
| **依赖关系** | 跨模块/跨服务调用是否正确？ |
| **风险点** | 是否有遗漏的安全、性能、合规风险？ |

#### 2.3 记录反馈

把 review 意见直接记录在 `docs/modules/{模块名}.md` 的末尾，或创建一个专门的 `docs/reviews/{模块名}-{日期}.md`：

```markdown
# Review 记录：{模块名} / {日期}

## 参与人
- 张三（技术负责人）
- 李四（产品经理）

## 主要反馈
1. 订单状态“待发货”应改为“待备货”——业务术语更新。
2. 支付回调幂等性机制需补充：使用 payment_no + 状态机判断。
3. 订单超时 30 分钟可能不准确，实际为 15 分钟。

## 修正动作
- [x] 更新术语表
- [x] 补充幂等性说明
- [x] 修改超时时间
```

#### 2.4 启用 Loop 模式 grill-deep-loop（推荐 review 前自我演练）

> **原则**：直接拉人开 review 会经常发生"问题没准备好，会议成了发散漫谈"。grill-deep-loop 让 AI **充当模拟业务方/架构师**，先帮你跑一遍杀伤力提问，捋出 review 必须解决的真问题。  
> **配套 skill**：`AI_工作流SOP_阶段3_Skill_grill-deep-loop.md`。

| 场景 | 推荐模式 |
|---|---|
| 快速捋 3-5 个关键问题 | grill-deep 即可 |
| review 会议前的自我演练 | **grill-deep-loop** |
| 业务方很忙，需精准提问 | **grill-deep-loop** |
| 接手陌生模块第一次 grill | **grill-deep-loop** |
| reverse-loop 产出后的最后验证 | **grill-deep-loop** |

**6 维度（GQ1–GQ6）**：边界条件 / 异常路径 / 事务与一致性 / 幂等与重试 / 并发与时序 / 缓存与历史数据。每维 ≥2 问。

**每条问题硬性要求**：
- 锚定到具体方法 / 字段 / 接口（禁止"如何保证 XX"空泛问法）
- 给出"不答的后果"
- 给出"期望答案空间"（YES/NO / 枚举 / 数值 / 需业务方决策）

**调用方式**：
- 通用 prompt：见 `AI_工作流SOP_阶段3_Skill_grill-deep-loop.md` § 5
- Claude Code 命令：`/grill-deep-loop order-timeout`
- Cursor：见 skill 文档 § 7

**停止条件**：GQ1–GQ8 全 ✅ 立即停 / 最多 2 轮自查 / 同一问题换皮算 1 条。

**输出衔接**：
- "优先级 5 问" → 直接进入 §2.2 review 议程
- "业务方专属问题" → review 会前发给业务方，避免临场发懵
- AI 自答不了的 GQ → 作为本节 §2.3 反馈的"待确认项"，进入 §4 共识检查清单

### 步骤 3：更新 `CLAUDE.md`（根 / 模块级 / 域级）

根据 review 结果，决定哪些信息应该进入 `CLAUDE.md`，哪些信息应该留在过程梳理文档中。

> **核心区分**：`CLAUDE.md` 是 AI 的**上下文配置文件**，告诉 AI 如何在这个项目/模块/域中工作；过程梳理文档（`docs/modules/`、`docs/domains/`、`docs/glossary.md`、`docs/entry-points.md`）是**业务知识记录**。
>
> `CLAUDE.md` 只保留对 AI 行为有直接约束的**长期稳定**信息；具体业务规则、流程细节、接口列表只在 `CLAUDE.md` 中以**指针**形式出现，不复制内容。

#### 写入 `CLAUDE.md` 的条件

- 长期稳定、跨功能通用的约定；
- 如果不写入，后续 AI 会话大概率会再次犯错；
- 属于团队红线或架构约束；
- 影响代码生成质量的结构性信息（包结构、分层、命名、测试策略）。

**示例**：
- ✅ 写入：`所有 Controller 必须返回统一响应体`、`订单状态变更必须通过 StateMachine`、`禁止跨层调用`；
- ✅ 写入：包结构约定、测试框架选择、外部接口调用方式；
- ✅ 写入：文档指针，如 `订单状态机定义见 docs/domains/订单域.md`。

#### 不写入 `CLAUDE.md`，留在过程梳理文档的条件

- 仅适用于该模块/域的业务规则；
- 可能会随需求频繁变化；
- 具体实现细节、接口参数、字段含义；
- 详细流程图、状态机、数据模型。

**示例**：
- ❌ 不写入：`订单超时 30 分钟自动取消`（具体业务规则，放在 `docs/domains/订单域.md`）；
- ❌ 不写入：`/api/v1/orders` 的字段列表（放在 `docs/entry-points.md`）；
- ❌ 不写入：`OrderStatus` 枚举的每个取值含义（放在 `docs/glossary.md`）。

#### 更新操作

```bash
# 1. 打开对应的 CLAUDE.md 编辑
#    - 根 CLAUDE.md：项目全局约定
#    - 物理模块级 CLAUDE.md：多模块项目时每个 Maven/Gradle 模块一个
#    - 域级 CLAUDE.md：单模块多业务域时每个业务域一个（docs/domains/ 或 docs/modules/）
# 2. 只修改相关章节，避免大改
# 3. 保存后检查行数是否仍然 ≤ 200（根）或 ≤ 150（模块级/域级）

wc -l CLAUDE.md
wc -l {模块}/CLAUDE.md
wc -l docs/domains/{域}.md
```

> **注意**：如果某条业务规则足够稳定且影响 AI 行为（例如“订单超时后必须触发补偿流程”），可以在域级 `CLAUDE.md` 中保留**高度概括**的一句话，并指向详细文档。不要把具体实现写进去。

### 步骤 4：生成共识检查清单

创建 `docs/consensus-checklist.md`（或放在模块文档内）：

```markdown
# 共识检查清单：{模块名}

## 业务共识
- [ ] 该模块的核心业务目标已被业务方确认
- [ ] 关键业务术语已统一并写入术语表
- [ ] 状态流转图已确认
- [ ] 边界条件（异常、空值、超时、重试）已覆盖

## 技术共识
- [ ] 事务边界和并发策略已明确
- [ ] 幂等性、补偿机制已明确
- [ ] 缓存与数据库一致性策略已明确
- [ ] 外部接口/消息契约已确认
- [ ] 依赖模块和调用关系已记录

## 文档共识
- [ ] 根 CLAUDE.md 已更新，且 ≤ 200 行
- [ ] 模块级/域级 CLAUDE.md 已更新（如适用），且 ≤ 150 行
- [ ] 模块文档/域文档已通过 review
- [ ] 术语表、入口地图已更新

## 进入正向迭代的条件
- [ ] 以上所有项已勾选
- [ ] 存在明确的 PRD 或 Issue
- [ ] 验收标准（AC）已定义
```

只有清单全部勾选，才允许进入阶段 4。

### 步骤 5：生成 PRD

使用 `/to-prd` skill：

```markdown
# 角色
你是一名产品经理 + 技术写手的混合角色。请根据以下需求描述和项目上下文，生成一份结构化的 PRD。

# 输入
## 需求描述
{粘贴业务方/产品经理的需求}

## 根 CLAUDE.md
{粘贴}

## 模块级/域级 CLAUDE.md（如有）
{粘贴}

## 模块/域文档
{粘贴 docs/modules/{模块名}.md 或 docs/domains/{域}.md 的核心部分}

# 输出格式
## 背景与目标
## 用户故事
## 功能范围（In Scope / Out of Scope）
## 业务流程
## 接口/入口定义
## 数据变更
## 非功能需求（性能、安全、稳定性）
## 验收标准（AC）
## 风险与依赖
## 待确认项
```

把生成的 PRD 保存到 `docs/prd/{功能名}.md`。

### 步骤 6：拆分为开发任务（Issue）

使用 `/to-issues` skill：

```markdown
# 角色
你是一名技术负责人。请把以下 PRD 拆分为可开发的任务列表，每个任务必须可在一个工作日内完成或在一个 PR 中合并。

# 输入
PRD：{粘贴 docs/prd/{功能名}.md}

## 约束
- 每个任务必须有明确的验收标准；
- 任务之间应尽量减少依赖；
- 优先拆出独立的领域模型/接口契约/测试任务；
- 如果涉及数据库变更，单独拆一个任务；
- 如果涉及外部接口变更，单独拆一个任务。

# 输出格式
| 任务编号 | 任务标题 | 类型 | 验收标准 | 依赖 | 预估工时 |
|---|---|---|---|---|---|
| ISSUE-001 | ... | 领域模型 | ... | 无 | 4h |
```

#### 录入 Issue 跟踪工具

把任务录入到团队使用的工具中：

```markdown
# GitHub Issues 示例
- [ ] ISSUE-001: 定义订单领域模型 (#101)
- [ ] ISSUE-002: 实现创建订单接口 (#102)
- [ ] ISSUE-003: 订单超时取消定时任务 (#103)
- [ ] ISSUE-004: 订单支付回调幂等性 (#104)
```

## 5. 常见问题

### Q1：业务方说没时间 review 怎么办？

**答**：至少把关键业务术语、状态流转、接口契约三项确认清楚。其他内容可以异步确认，但必须在进入正向迭代前以书面形式（Issue/PRD 评论）留痕。

### Q2：AI 生成的 PRD 太技术化怎么办？

**答**：让 AI 先按“产品经理视角”生成 PRD，再让 AI 按“技术实现视角”补充实现章节。不要混在一起生成。

### Q3：共识检查清单全部勾选后，需求又变了怎么办？

**答**：重新走共识确认阶段。如果变更很小，可以只做最小范围的更新；如果变更影响架构或状态流转，必须重新 review。

### Q4：一个功能涉及多个模块，怎么拆任务？

**答**：先确认每个模块的边界和接口契约。然后为每个模块创建独立的任务，最后用一个“集成验收”任务把它们串起来。
