# reverse-loop —— 逆向理解模块（AI 执行版）

> **何时启动**：用户要"梳理 / 逆向 / 理解 / 还原 / 看懂"一个模块。
> **禁止**：自由发挥；不读代码就开始写；编造行号。

---

## 输入清单（缺即停）

| 项 | 必需 | 缺失时动作 |
|---|---|---|
| 模块名 / 边界 | 必需 | 反问 "要分析哪个模块？" |
| 关键代码（Controller/Service/Mapper/Entity 各至少 1） | 必需 | 用 Read/Glob 取，或要用户指定路径 |
| 根 CLAUDE.md | 强烈建议 | 缺时标记"无项目级约束" |
| 数据库 schema | 强烈建议 | 缺时数据模型字段全打 `[待验证]` |

---

## 输出文件（2 个，缺一不可）

| 文件 | 路径模板 | 包含段落 | 寿命 |
|---|---|---|---|
| **模块文档** | `docs/modules/{模块名}.md` | 段一：理解范围 / 业务职责 / 接口与入口 / 流程 / 数据模型 / 风险 | **长期**（团队维护，下次跑 reverse-loop 时覆盖） |
| **Loop 报告** | `docs/reports/reverse-{模块名}-{YYYY-MM-DD}.md` | 段二：自查表 + 段三：需人工确认清单 + 段四：迭代信息 | **短期**（任务归档） |

如果项目没有 `docs/` 目录，落到模块根目录下相似命名（`{模块名}.md` + `reverse-{模块名}-report.md`）。

---

## 流程（6 步，禁止跳步）

1. **扫**：用 Glob 列出模块根目录所有 Java 文件，输出文件清单
2. **分批读**：按 入口 Controller → Service → Mapper → Entity → 外部 Client 顺序 Read
3. **出初稿**：按下方"输出格式"分别写 2 个文件的第一版
4. **自查**：按 PASS 条件 AC1–AC6 填自查表（写入 Loop 报告），每条给证据
5. **修正**：仅对 ⚠️/❌ 项改一轮（最多 1 次）
6. **终输出**：确认 2 个文件都已生成，跑校验脚本

---

## PASS 条件（自查标准，严格解释）

### AC1 业务职责
- ✅ 满足全部：①句子数 ≥ 2（1 句概述 + 1 段详述）②含 ≥ 2 个 `文件名.java:行号` 引用 ③引用的行号已用 Read 验证过
- ⚠️ 句数够但引用 < 2
- ❌ 只写了概念无代码引用

### AC2 接口与入口
- ✅ 表格列含：入口 / 类型(HTTP/Kafka/Schedule) / 方法 / 路径 / 关键参数 / 事务边界
- ⚠️ 表格缺事务边界列
- ❌ 漏掉任一类型入口（如只列 HTTP 漏 @Scheduled）

### AC3 核心流程
- ✅ Mermaid 图 + 每个节点配方法名（如 `OrderService.cancel()`）
- ⚠️ 有图但部分节点是中文描述非方法名
- ❌ 只有文字描述无图

### AC4 数据模型
- ✅ 表格列含：实体类 / 表名 / 主键 / 状态字段及全部枚举值 / 索引（如知）
- ⚠️ 状态字段未列全部枚举
- ❌ 实体未对应到具体表名

### AC5 风险与异味
- ✅ ≥ 3 条，每条含 `文件:行号` + 风险类型（并发/事务/空指针/性能等）
- ⚠️ 有 3 条但其中 1 条无行号
- ❌ < 3 条 或 全部空泛描述

### AC6 不确定项标记
- ✅ 所有"可能 / 似乎 / 大概 / 应该是 / 推测"所在的段落开头有 `[待验证]` 标签
- ⚠️ 个别段落漏标（≤ 2 段）
- ❌ 通篇没有 `[待验证]` 但又有不确定描述

## 禁用模式（出现即扣分）

| 禁用写法 | 应改为 |
|---|---|
| "该模块负责订单相关功能" | 必须带文件:行号 |
| "可能存在并发问题"（独立成句） | "[待验证] OrderService.java:88 未加锁，可能存在并发问题" |
| 状态字段只列 "0/1/2" 无含义 | "0=待支付, 1=已支付, 2=已取消" |
| "建议优化" | 必须说优化什么、改哪里 |
