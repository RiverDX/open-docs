# doc-rot-loop —— 文档腐化巡检（AI 执行版）

> **何时启动**：用户要"巡检文档 / 检查腐化 / doc rot / 文档健康度"。  
> **禁止**：堆 P2 nitpick；空证据；P0 > 5 条；同类问题不合并。

---

## 输入清单（缺即停）

| 项 | 必需 | 缺失时动作 |
|---|---|---|
| 根 CLAUDE.md + 模块/域级 CLAUDE.md | 必需 | 用 Glob 找：`**/CLAUDE.md` |
| docs/entry-points.md | 必需 | Read 取，缺则在 DR2 全部标 P0 |
| docs/glossary.md | 必需 | Read 取 |
| docs/modules/*.md | 必需 | Glob + Read |
| docs/ai-skills/*.md（或 AI版/loops/） | 强烈建议 | Glob |
| 近 N 天 git log | 必需 | Bash 跑 `git log --since="30 days ago" --oneline` |
| 近 N 天文件变更列表 | 必需 | `git log --since="30 days ago" --name-only` |
| 机械检查脚本输出 | 建议 | 跑 `python scripts/check_*.py` |

---

## 流程（6 步，禁止跳步）

1. **读输入**：把所有 CLAUDE.md / docs / git log 读到工作上下文
2. **建变更主线**：从 git log 抽出近期主要变更（新增接口、删除接口、新增模块、状态枚举变化）
3. **按 DR 逐类查**：DR1 → DR2 → DR3 → DR4 顺序，每类必须给结果
4. **打标**：每条腐化打 P0/P1/P2 + 修复成本 + 修复方式 + 责任人方向
5. **自查 + 合并**：按 PASS 条件 AC1–AC6 自查；P0 > 5 → 立即合并同类或降级
6. **终输出**：4 段（健康度总览 + 腐化清单 + 自查表 + 行动清单）

---

## PASS 条件（自查标准，严格解释）

### AC1 DR1–DR4 全覆盖
- ✅ 4 类都被检查，无问题的也标 "✅ 健康"
- ⚠️ 漏 1 类
- ❌ 漏 ≥ 2 类

### AC2 每条腐化有文件路径 + 证据片段
- ✅ 100% 含 `文件:行号` + 引用 ≤ 5 行
- ⚠️ ≥ 90%
- ❌ < 90%

### AC3 优先级符合判定标准
- ✅ 所有 P0 都符合"误导开发者"或"已废弃但仍推荐"或"关键约束失效"
- ⚠️ 1-2 条 P0 应降级
- ❌ ≥ 3 条 P0 应降级

### AC4 P0 ≤ 5 条
- ✅ P0 ≤ 5
- ⚠️ P0 = 6-7（需在自查中说明）
- ❌ P0 ≥ 8（强制合并 / 降级）

### AC5 每条标修复方式
- ✅ 100% 标 [可脚本] / [需人工] / [需业务方] + 修复成本 [N 分钟] / [N 小时]
- ⚠️ ≥ 90%
- ❌ < 90%

### AC6 健康度评分有依据
- ✅ 每类评分都按公式 + 写出扣分项
- ⚠️ 评分有公式但没列扣分项
- ❌ 凭感觉打分

---

## DR1–DR4 检查项明细

### DR1 CLAUDE.md & 全局配置
- 根 CLAUDE.md 行数是否超 200
- 模块级 / 域级 CLAUDE.md 行数是否超 150
- CLAUDE.md 引用的文件 / 章节是否仍存在
- CLAUDE.md 中的约束是否还在被遵守（如"禁止 Lombok @Data" 但 diff 中有）

### DR2 入口地图 & API 契约
- 代码中所有 @*Mapping 是否都在 entry-points.md
- entry-points.md 中的接口是否都还在代码中
- 新增的 @KafkaListener / @RabbitListener / @Scheduled 是否入 entry-points

### DR3 模块文档 & 术语表
- docs/modules/*.md 中描述的类 / 方法 / 字段是否仍存在
- docs/glossary.md 中的术语是否仍被代码使用（grep）
- 近期新增的业务术语 / 状态枚举是否进术语表

### DR4 AI Skill 模板
- docs/ai-skills/*.md（或 AI版/loops/）中引用的 CLAUDE.md 章节是否还在
- skill 的硬性约束是否需要根据近期项目变化调整

---

## 优先级判定（严格）

### P0（本周修）
- 文档与代码冲突 + 会误导新人或 AI（如术语表说"订单状态 0-2" 但代码已有 9）
- 已废弃接口仍在 entry-points 推荐使用
- CLAUDE.md 中关键约束已被普遍违反

### P1（本月修）
- 文档与代码偏差较小但需修正
- 新增功能未同步术语表 / 入口地图
- skill prompt 引用了已删除的章节

### P2（下季度前修）
- 可读性 / 行文风格不统一
- 可优化但不影响 AI 工作流
- 新模型下的 prompt 可微调

---

## 健康度评分公式（强制使用）

```
单类健康度 = 100 - (P0 × 15) - (P1 × 5) - (P2 × 1)
加权总分 = DR1×0.3 + DR2×0.25 + DR3×0.3 + DR4×0.15
```

参考：
- ≥ 90 优秀
- 75-89 良好
- 60-74 一般
- < 60 差（建议暂停新功能开发，先修文档）

## 禁用写法（出现即扣分）

| 禁用 | 替换 |
|---|---|
| "总体健康度：良好" 无评分 | 必须给 0-100 数字 + 公式扣分项 |
| "建议更新文档" | "建议在 docs/entry-points.md:42 后补一行：`/api/v1/payment-callback` | POST | 支付回调 |" |
| 50 条 P2 nitpick | 同类合并；ROI 低的删除 |
| P0 全是"建议改进" | P0 必须是"误导"级别，否则降级 |
