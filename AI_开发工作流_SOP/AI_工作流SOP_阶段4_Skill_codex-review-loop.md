# Skill：codex-review-loop —— 代码预审查的 Loop 版本

> **定位**：把 SOP 阶段 4 中的 `codex-review`（单次提问版）升级为 **Loop 版本** —— 让 AI 自己按 6 大审查维度循环检查、严重度分级、自查证据，最终输出一份「人工 reviewer 可以直接照单全收的预审报告」。  
> **关联文件**：`AI_工作流SOP_阶段4_正向迭代_操作手册.md`、`AI_工作流SOP_阶段4_Skill_tdd-loop.md`、`AI_工作流SOP_阶段2_Skill_reverse-loop.md`  
> **理论基础**：Loop Engineering 4 模块（目标 / 步骤 / 检查 / 停止）。  
> **版本**：V1.0（2026-06-23）

---

## 1. 目标

让 AI 在提交 PR 前先对 diff 做一次**结构化、可追溯**的预审：每条问题都带文件:行号 + 严重度 + 建议修复 + 自评置信度。  
不再出现「AI 列了 30 条建议，人工 reviewer 不知道哪条该看」的情况。

与 `codex-review`（单次版）的差异：

| 对比项 | codex-review（旧） | codex-review-loop（新） |
|---|---|---|
| 输出形式 | 一张大表，问题混在一起 | 按 严重 / 建议 / 提示 三级 + 6 维度组织 |
| 严重度判断 | 全靠 AI 个人发挥 | 强制按 SEV1-3 标准分级 |
| 证据要求 | 经常给"建议优化"但没说哪里坏 | 必须给文件:行号 + 引用代码片段 |
| 自评 | 无 | 每条问题 AI 自评置信度（高/中/低） |
| 越界控制 | 无（可能罗列上百条 nitpick） | 最多 2 轮自查，超时即停 |
| 适合人群 | 已熟悉项目的高级开发者 | 小白也能直接根据"严重"标签处理 |

---

## 2. 触发条件

满足任一条即可启用 codex-review-loop 而不是 codex-review：

- 准备提 PR 前的最后一道关；
- diff > 200 行，单次审查容易抓不住重点；
- 改动涉及核心模块（订单、支付、用户、权限）；
- 团队 reviewer 比较忙，希望 AI 先过滤明显问题；
- 跑过一次 codex-review 但建议太散，需要重新组织。

不需要 Loop 的场景（用 codex-review 即可）：

- diff < 50 行的小改；
- 只改了测试或文档；
- 已经经过人工 review，AI 只是补充意见。

## 3. 输入

| 输入项 | 是否必需 | 说明 |
|---|---|---|
| Git diff | 必需 | `git diff main...HEAD` 或 PR 链接 |
| PRD / Issue | 必需 | 确认 diff 是否实现了 AC |
| 根 `CLAUDE.md` | 必需 | 全局约束 |
| 模块级 `CLAUDE.md` | 可选 | 模块约定 |
| 现有测试报告（如有 tdd-loop 产出） | 强烈建议 | 用于 RC3 维度 |
| 提交信息 / commit message | 建议 | 帮助 AI 理解意图 |

---

## 4. 输出

跑完一次 Loop，AI 必须输出 **4 段**：

### 4.1 总览摘要

```markdown
| 项 | 数量 |
|---|---|
| SEV1 严重（阻断合并） | N |
| SEV2 建议（应修复） | N |
| SEV3 提示（可选） | N |
| 总置信度（高/中/低问题占比） | ... |
| 建议人工特别关注 | top 3 文件 / 行号 |
```

### 4.2 按维度分组的问题列表

按 6 个维度组织，每条含：严重度 / 位置 / 问题 / 建议 / 置信度。

### 4.3 自查表（核心新增产物）

```markdown
| 审查维度 | 评分 | 证据 / 卡点 |
|---|---|---|
| RC1 PRD 一致性（与 AC 对比） | ✅ / ⚠️ / ❌ | ... |
| RC2 CLAUDE.md 约定遵循 | ✅ / ⚠️ / ❌ | ... |
| RC3 测试覆盖与质量 | ✅ / ⚠️ / ❌ | ... |
| RC4 并发 / 事务 / 空指针 / 安全 | ✅ / ⚠️ / ❌ | ... |
| RC5 命名 / 分层 / 重复代码 | ✅ / ⚠️ / ❌ | ... |
| RC6 文档同步（CLAUDE.md / docs/） | ✅ / ⚠️ / ❌ | ... |
```

### 4.4 需人工确认清单

```markdown
- [ ] 业务规则是否符合预期：xxx
- [ ] 性能影响是否可接受：xxx
- [ ] 是否需要 ADR / 架构决策记录：xxx
```

## 5. 通用 Prompt（核心模板，小白可直接复制粘贴）

```markdown
# 角色
你是一名严格的 Java / SpringBoot 代码审查员，目标是用 Loop 方式对 PR diff 做结构化预审。
你的工作不是"找毛病越多越好"，而是"帮人工 reviewer 把注意力集中到真问题上"。

# 任务
对以下 diff 做预审，并输出结构化报告。

# 流程（必须按此顺序，不要跳步）
1. 通读 diff，识别本次改动覆盖的功能边界
2. 按下方 6 个审查维度 (RC1–RC6) 逐项检查
3. 每条问题打上严重度（SEV1/SEV2/SEV3）和置信度（高/中/低）
4. 按下方"验收标准"自查（是否每条都给了证据、严重度是否合理）
5. 对自评 ⚠️/❌ 的维度补充检查（最多 2 轮）
6. 输出总览摘要 + 问题列表 + 自查表 + 需人工确认清单

# 6 个审查维度（必须每个都过一遍，没有问题也要在自查表里 ✅）
- RC1 PRD 一致性：diff 是否实现了 PRD 中的全部 AC？有没有偏离？
- RC2 CLAUDE.md 约定：是否违反全局/模块约束（分层、包结构、禁用清单）？
- RC3 测试覆盖与质量：新代码是否有测试？是否覆盖边界条件？断言是否可证伪？
- RC4 并发 / 事务 / 空指针 / 安全：是否有数据竞争、N+1、未捕获异常、SQL 注入、敏感日志？
- RC5 命名 / 分层 / 重复代码：方法/变量名是否清晰？是否破坏分层？是否有可提取的公共逻辑？
- RC6 文档同步：是否更新了 CLAUDE.md / docs/glossary.md / docs/entry-points.md？

# 严重度分级标准（必须严格遵守）
- SEV1（严重，阻断合并）：bug、安全漏洞、违反核心约束、必然导致线上事故的问题
- SEV2（建议，应在本 PR 修复）：可能导致问题但非必然、违反一般约定、明显的代码异味
- SEV3（提示，可下个 PR 跟进）：可读性、命名、轻微重复、文档优化

# 验收标准（自查表用，每项必须有证据）
- AC1 每条问题都有 文件:行号 + 引用 ≤5 行代码片段
- AC2 每条问题都有 SEV 等级 + 自评置信度
- AC3 SEV1 问题不超过 5 条（超过说明分级偏严，需要降级或合并）
- AC4 RC1–RC6 每个维度都被检查过（即使 ✅ 也要在自查表标明）
- AC5 总览摘要的"建议人工特别关注"不超过 3 条（避免摊薄注意力）
- AC6 不确定的判断必须降级为 SEV3 + 置信度"低"，禁止靠猜测打 SEV1

# 停止条件
- 全部 AC1–AC6 评为 ✅ → 立即停止
- 已自查修改 2 轮仍有 ⚠️/❌ → 停止，需人工确认清单中明确"哪些维度信息不足"
- 严禁为"显得审查更专业"而堆砌 SEV3 nitpick

# 输入
## PRD / Issue
{粘贴}

## 根 CLAUDE.md
{粘贴}

## 模块级 CLAUDE.md
{粘贴；无写"无"}

## Diff
```diff
{粘贴 git diff 内容}
```

## 测试报告（如有 tdd-loop 产物）
{粘贴}

## Commit message
{粘贴}

# 输出格式（严格遵循）
## 一、总览摘要
| 项 | 数量 / 说明 |
|---|---|
| SEV1 | N |
| SEV2 | N |
| SEV3 | N |
| 建议人工特别关注 | 文件:行号 ×3 |

## 二、按维度分组的问题列表

### RC1 PRD 一致性
| SEV | 位置 | 问题 | 建议 | 置信度 |
|---|---|---|---|---|

### RC2 CLAUDE.md 约定
...

### RC3 测试覆盖与质量
...

### RC4 并发 / 事务 / 空指针 / 安全
...

### RC5 命名 / 分层 / 重复代码
...

### RC6 文档同步
...

## 三、自查表
| AC | 评分 | 证据 / 说明 |
|---|---|---|
| AC1 位置+引用 | ✅/⚠️/❌ | ... |
| AC2 SEV+置信度 | ✅/⚠️/❌ | ... |
| AC3 SEV1 ≤ 5 | ✅/⚠️/❌ | ... |
| AC4 RC1–RC6 全覆盖 | ✅/⚠️/❌ | ... |
| AC5 重点 ≤ 3 | ✅/⚠️/❌ | ... |

## 6. Claude Code 适配

`.claude/commands/codex-review-loop.md`：

```markdown
---
description: 用 Loop 方式对当前分支的 diff 做结构化预审
argument-hint: <可选：对比基准分支，默认 main>
---

请用 codex-review-loop 模式审查当前分支 PR。

# 步骤
1. 跑 `git diff ${1:-main}...HEAD` 取 diff
2. 读 docs/prd/ 下对应的 PRD（从 commit message 或分支名推断）
3. 读 CLAUDE.md（根 + 模块）
4. 按 docs/ai-skills/codex-review-loop.md 的 RC1–RC6 维度检查
5. 最多自查 2 轮
6. 报告写入 docs/review-reports/PR-{时间戳}.md
7. SEV1 问题在对话末尾红色高亮列出
```

使用方式：

```
/codex-review-loop
/codex-review-loop develop  # 对比 develop 分支
```

## 8. 注意事项 / 常见陷阱

### 8.1 AI 做代码审查最容易掉的 5 个坑

| 坑 | 表现 | 应对 |
|---|---|---|
| **SEV 滥用** | 把"命名不够好"打成 SEV1 | 严格按分级标准，AC3 上限 5 条 |
| **空建议** | "建议优化"但没说怎么改 | AC1 强制要求建议必须可执行 |
| **重复警告** | 同一类问题在 6 个位置各报一次 | prompt 要求同类问题合并为一条 |
| **过度自信** | 编造 PRD 没说的"应该" | AC6 要求不确定的降级 + 置信度"低" |
| **没看 PRD** | 只看代码，忽略业务上下文 | RC1 强制以 AC 为对照 |

### 8.2 项目可加的自定义 RC（按需追加）

- **RC7 性能**：N+1 查询、循环内调用、未加索引的查询；
- **RC8 安全**：SQL 注入、XSS、敏感日志、密钥硬编码；
- **RC9 国际化**：硬编码中文/英文字符串；
- **RC10 可观测性**：缺日志、缺埋点、缺 traceId 透传。

### 8.3 SEV1 问题特别处理

- **每条 SEV1 必须 PR 内修复**：否则不进入人工审查；
- **SEV1 数量 > 5**：先怀疑分级太严，与人工 reviewer 沟通后降级或重审；
- **SEV1 置信度低**：升级为"需人工立即确认"，不要直接打回 PR。

### 8.4 与其他 Skill 的衔接

- **codex-review-loop 跑完后** → SEV1 在本 PR 修；SEV2 评估是否在本 PR 修；SEV3 转为 issue 跟进；
- **与 tdd-loop 联动**：RC3 直接读 tdd-loop 自查表的 TC6 评分；
- **与 reverse-loop 联动**：RC1 引用 reverse-loop 输出的接口契约部分。

## 9. 一次最小可用示例

**输入**：

```
分支：feat/order-timeout（基于阶段 4 操作手册的订单超时取消功能）
diff：120 行（OrderTimeoutJob.java +60, OrderService.java +40, OrderTimeoutJobTest.java +20）
PRD：docs/prd/order-timeout.md
```

**期待 AI 输出（节选）**：

```markdown
## 一、总览摘要
| 项 | 数量 / 说明 |
|---|---|
| SEV1 | 1 |
| SEV2 | 3 |
| SEV3 | 4 |
| 建议人工特别关注 | OrderService.java:88（事务边界）|

## 二、按维度

### RC1 PRD 一致性
| SEV | 位置 | 问题 | 建议 | 置信度 |
|---|---|---|---|---|
| SEV2 | OrderTimeoutJob.java:42 | PRD AC3 要求重试 3 次后写日志，代码只重试 1 次 | 改用 @Retryable(maxAttempts=3) 或显式 for 循环 | 高 |

### RC4 并发 / 事务 / 空指针 / 安全
| SEV | 位置 | 问题 | 建议 | 置信度 |
|---|---|---|---|---|
| SEV1 | OrderService.java:88 | cancelOrder 内先 update order 状态、再调外部 InventoryClient，外部失败会导致订单状态已变更但库存未释放 | 用 @Transactional + 本地表事件 + 异步重试，或先调外部再 update | 高 |

### RC5 命名 / 分层
| SEV | 位置 | 问题 | 建议 | 置信度 |
|---|---|---|---|---|
| SEV3 | OrderTimeoutJob.java:15 | 类名 OrderTimeoutJob 偏笼统，建议 CancelTimeoutOrderJob | 重命名 | 中 |

## 三、自查表
| AC | 评分 | 证据 |
|---|---|---|
| AC1 位置+引用 | ✅ | 所有问题均含文件:行号 |
| AC2 SEV+置信度 | ✅ | 全部标注 |
| AC3 SEV1 ≤ 5 | ✅ | 仅 1 条 SEV1 |
| AC4 RC1–RC6 全覆盖 | ✅ | 6 维度均检查；RC6 文档同步 ✅ 无需修改 |
| AC5 重点 ≤ 3 | ✅ | 仅推 1 条事务问题 |
| AC6 不确定降级 | ✅ | RC5 命名问题置信度"中" |

迭代轮次：1
是否达到停止条件：是

## 四、需人工确认清单
- [ ] SEV1 事务边界问题需要架构讨论（是否引入本地消息表）
- [ ] PRD AC3 重试 3 次是否含首次调用？需向 PO 确认
```

这就是一次合格的 codex-review-loop 输出 —— **不是"罗列建议"，而是"分层、有据、可执行"的预审报告**。
