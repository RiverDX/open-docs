# Skill：doc-rot-loop —— 文档腐化巡检的 Loop 版本

> **定位**：把阶段 5 中的 `doc-rot-inspection`（单次提问版）升级为 **Loop 版本** —— 强制按 4 大文档类型 × P0/P1/P2 优先级生成发现，每条腐化必须给"证据 + 自动可修复标记 + 责任归属"，避免巡检报告退化成"AI 一通模糊吐槽，没人能落地"。  
> **关联文件**：`AI_工作流SOP_阶段5_持续演化_操作手册.md` § 步骤 2、所有上游 Loop skill  
> **理论基础**：Loop Engineering 4 模块（目标 / 步骤 / 检查 / 停止）。  
> **版本**：V1.0（2026-06-23）

---

## 1. 目标

让 AI 月度巡检的输出**直接可转 issue**，每条问题都能被某个具体人接住：含腐化类型、优先级、证据、修复成本估计、是否可自动修复、责任人建议。  
不再出现「报告写得很漂亮，但没人知道下周该做什么」的情况。

与 `doc-rot-inspection`（单次版）的差异：

| 对比项 | doc-rot-inspection（旧） | doc-rot-loop（新） |
|---|---|---|
| 检查项 | 7 条文字清单，混在一起 | 按 4 类（DR1–DR4）分组 |
| 优先级 | P0/P1/P2 标签但无标准 | 强制定义 P0/P1/P2 判定规则 |
| 证据要求 | 经常"建议更新某节" | 必须给文件路径 + 行号 + 引用片段 |
| 可操作性 | 报告读完不知道下一步 | 每条标"可脚本修复 / 需人工 / 需业务方" |
| 越界控制 | 无 | 最多 2 轮自查，超时即停 |
| 健康度评分 | 主观（"良好 / 一般"） | 4 类各打 0-100 + 加权总分 |

---

## 2. 触发条件

满足任一条即用 doc-rot-loop 替代 doc-rot-inspection：

- 月度定期巡检（默认每月一次）；
- PR 频次较高（> 30 PR/月）的项目，建议双周一次；
- 季度 OKR review 前；
- 团队接手老项目首次盘点；
- 发现某次 PR 出现"文档与代码严重不一致"事故后的事后复盘。

不需要 Loop 的场景（用 doc-rot-inspection 即可）：

- 只想快速对一个文件做一致性检查；
- 临时性的代码 review 顺带瞄一眼文档；
- 已知腐化点的针对性修复，无需全量扫描。

## 4. 输出

跑完一次 Loop，AI 必须输出 **4 段**：

### 4.1 总览健康度

```markdown
| 文档类 | 健康度 (0-100) | P0 数 | P1 数 | P2 数 | 主要问题 |
|---|---|---|---|---|---|
| DR1 CLAUDE.md & 全局配置 | ... | ... | ... | ... | ... |
| DR2 入口地图 & API | ... | ... | ... | ... | ... |
| DR3 模块文档 & 术语表 | ... | ... | ... | ... | ... |
| DR4 AI Skill 模板 | ... | ... | ... | ... | ... |
| **加权总分** | ... | | | | |
```

权重建议：DR1 30% / DR2 25% / DR3 30% / DR4 15%。

### 4.2 按类分组的腐化清单（核心产物）

每条腐化含：**位置 / 优先级 / 问题描述 / 证据片段 / 修复建议 / 修复成本 / 是否可脚本修复 / 责任人建议**。

### 4.3 自查表

```markdown
| 验收标准 | 评分 | 证据 |
|---|---|---|
| AC1 DR1–DR4 四类都检查过 | ✅/⚠️/❌ | ... |
| AC2 每条腐化都有文件路径 + 证据片段 | ✅/⚠️/❌ | ... |
| AC3 每条都标 P0/P1/P2 且符合判定标准 | ✅/⚠️/❌ | ... |
| AC4 P0 条目不超过 5 条 | ✅/⚠️/❌ | ... |
| AC5 每条都标"可脚本/人工/业务方" | ✅/⚠️/❌ | ... |
| AC6 健康度评分有具体扣分理由 | ✅/⚠️/❌ | ... |
```

### 4.4 行动清单（issue 草稿）

```markdown
## 本月必做（P0，建议本周完成）
- [ ] DOC-001 ...

## 本月应做（P1，建议本月完成）
- [ ] DOC-005 ...

## 后续跟进（P2，下季度前完成）
- [ ] DOC-012 ...

## 已废弃但保留供历史参考
- DOC-XXX ...
```

## 5. 通用 Prompt（核心模板，小白可直接复制粘贴）

```markdown
# 角色
你是该 Java / SpringBoot 项目的知识管理专员，目标是用 Loop 方式做一次月度文档腐化巡检。
你的工作不是"找问题越多越好"，而是"把真问题分清优先级，并交到该交的人手里"。

# 任务
对项目当前的 CLAUDE.md / docs / AI skill 与近期代码变更做一致性检查，输出可直接转 issue 的腐化报告。

# 流程（必须按此顺序，不要跳步）
1. 通读输入材料，理解最近一个月的代码变更主线
2. 按 DR1–DR4 四类分别检查（每类必须给出结果，即使健康）
3. 对每条腐化打优先级（P0/P1/P2）+ 修复成本 + 可修复方式
4. 按下方"验收标准"自查
5. 对 ⚠️/❌ 项补充检查（最多 2 轮）
6. 输出健康度评分 + 腐化清单 + 自查表 + 行动清单（issue 草稿）

# 4 类文档（必须每类检查）
- DR1 CLAUDE.md & 全局配置：根 CLAUDE.md / 模块级 CLAUDE.md，引用的文件是否存在、行数是否超限、约束是否仍在生效
- DR2 入口地图 & API 契约：docs/entry-points.md vs 代码中 @*Mapping、@KafkaListener、@Scheduled 等
- DR3 模块文档 & 术语表：docs/modules/*.md 中描述的类/方法/字段是否仍存在；docs/glossary.md 中术语是否仍被代码使用
- DR4 AI Skill 模板：docs/ai-skills/*.md 中的 prompt 是否适配当前模型、引用的 CLAUDE.md 章节是否还在

# 优先级判定标准（严格遵守）
- P0（本周修）：文档与代码冲突且会误导开发者；已废弃接口仍在 entry-points 推荐使用；CLAUDE.md 中关键约束已无人遵守
- P1（本月修）：文档与代码偏差较小但需修正；新增功能未同步到术语表/入口地图；skill prompt 引用了已删除的章节
- P2（下季度前修）：可读性问题；行文风格不统一；可优化但不影响 AI 工作流；新模型下的 prompt 可微调

# 输出硬性约束（每条腐化必须满足）
- 必须给具体文件路径 + 行号 + 引用 ≤5 行的代码/文档片段
- 必须明确分类：DR1 / DR2 / DR3 / DR4
- 必须打优先级：P0 / P1 / P2
- 必须标修复方式：[可脚本] / [需人工] / [需业务方]
- 必须给修复成本：[5 分钟] / [30 分钟] / [2 小时以上]
- 必须建议责任人方向（不一定指名）：[本模块开发] / [架构组] / [产品]
- P0 条目总数不超过 5 条（超过说明分级偏严，需降级或合并）

# 停止条件
- 全部 AC1–AC6 评为 ✅ → 立即停止
- 已自查修改 2 轮仍有 ⚠️/❌ → 停止，需人工确认清单中明确"哪类信息不足"
- 严禁为凑数量而堆砌 P2 nitpick；同类问题应合并为一条

description: 用 Loop 方式做月度文档腐化巡检

---

请用 doc-rot-loop 模式做文档腐化巡检。

# 步骤
1. 读取根 CLAUDE.md、所有模块级 CLAUDE.md
2. 读取 docs/entry-points.md、docs/glossary.md、docs/modules/*.md、docs/ai-skills/*.md
3. 跑机械检查脚本：
   - python scripts/check_claude_md_refs.py
   - python scripts/check_api_doc_sync.py
4. 取 git log --since="${1:-30 days ago}" --oneline
5. 取 git log --since="${1:-30 days ago}" --name-only | sort -u
6. 按 docs/ai-skills/doc-rot-loop.md 的 DR1–DR4 检查
7. 最多自查 2 轮
8. 写入 docs/reports/doc-rot-{时间戳}.md
9. P0 条目同时写入 GitHub Issues（如已配置 gh cli）
```

使用方式：

```
/doc-rot-loop           # 默认 30 天
/doc-rot-loop "90 days ago"  # 季度巡检
```

## 8. 注意事项 / 常见陷阱

### 8.1 AI 做文档腐化巡检最容易掉的 5 个坑

| 坑 | 表现 | 应对 |
|---|---|---|
| **过度报告** | 50 条 P2，把团队淹没 | AC4 强制 P0 ≤ 5，同类合并 |
| **空证据** | "建议更新某节"但不说哪行 | AC2 强制带行号 + 片段 |
| **错失重大变更** | 漏掉新增的微服务 / 新建的 module | 强制读 git log --name-only |
| **业务/技术不分** | 让架构组去回答业务规则变化 | AC5 强制标"业务方" |
| **健康度评分玄学** | 凭感觉打 80 分 | AC6 强制扣分理由（"DR3 有 3 个 P0 → 扣 20 分"）|

### 8.2 项目可加的自定义 DR（按需追加）

- **DR5 数据库 schema**：DDL 与 Entity 不一致、新增字段未在文档体现；
- **DR6 配置项**：application.yml 新增项未在 README / CLAUDE.md 说明；
- **DR7 依赖版本**：pom.xml / build.gradle 升级后文档未同步；
- **DR8 ADR**：架构决策记录是否仍反映现状。

### 8.3 P0 条目的特别处理

- **本周必做**：建议直接挂到团队周会议程，分配负责人；
- **超过 5 条**：先怀疑分级太严，与团队 lead 沟通后降级或合并同类；
- **如果连续两个月有同一类 P0** → 不只是文档问题，是流程问题，需要在阶段 5 加自动化检查脚本。

### 8.4 健康度评分建议公式

```
单类健康度 = 100 - (P0数 × 15) - (P1数 × 5) - (P2数 × 1)
加权总分 = DR1×0.3 + DR2×0.25 + DR3×0.3 + DR4×0.15
```

参考标准：
- ≥ 90：优秀，仅微调
- 75-89：良好，按计划处理
- 60-74：一般，需要重点投入
- < 60：差，应暂停部分新功能开发，先修文档

### 8.5 与其他 Skill 的衔接

- **DR2 入口地图问题** → 触发 reverse-loop 重新梳理该模块；
- **DR3 模块文档问题** → 触发 reverse-loop 局部更新；
- **DR4 skill 问题** → 触发 skill 季度 review；
- **DR1 CLAUDE.md 问题** → 直接更新 + 写 `docs/claude-md-changelog.md`；
- **每月巡检 P0 数** → 进 SOP §9 度量指标的"文档腐化检出率"。

## 9. 一次最小可用示例

**输入**（模拟）：
