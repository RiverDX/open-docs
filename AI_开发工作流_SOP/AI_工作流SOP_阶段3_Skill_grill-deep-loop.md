# Skill：grill-deep-loop —— 深度反向追问的 Loop 版本

> **定位**：把 `grill-deep`（单次提问版）升级为 **Loop 版本** —— 强制 AI 按 6 大风险维度提问，每维至少 2 问，自查"是否问到了关键真问题"，避免 grill 退化成"问 30 个不痛不痒的问题"。  
> **关联文件**：`AI_工作流SOP_阶段2_逆向理解_操作手册.md` § 步骤 4、`AI_工作流SOP_阶段3_共识确认_操作手册.md` § 步骤 2、`AI_工作流SOP_阶段2_Skill_reverse-loop.md`  
> **理论基础**：Loop Engineering 4 模块（目标 / 步骤 / 检查 / 停止）+ 风险驱动提问法。  
> **版本**：V1.0（2026-06-23）

---

## 1. 目标

让 AI 在 grill 阶段输出**有杀伤力的问题**，而不是「请说一下您的需求」式的客套问题。  
每个问题必须：定位到具体场景 + 揭示具体风险 + 有可回答的答案空间 + 标出未答的后果。

与 `grill-deep`（单次版）的差异：

| 对比项 | grill-deep（旧） | grill-deep-loop（新） |
|---|---|---|
| 问题来源 | AI 自由发挥 | 强制按 GQ1–GQ6 六维生成 |
| 维度覆盖 | 容易偏向某一类（如全是边界条件） | 每维度 ≥2 问，缺一不可 |
| 问题质量 | 经常空泛（"如何保证一致性"） | 强制锚定到具体方法 / 字段 / 接口 |
| 答案的"杀伤力" | AI 不评估问题价值 | 每条问题 AI 自评"如果不答，可能导致什么后果" |
| 越界控制 | 无 | 最多 2 轮自查，超时即停 |
| 适合人群 | 已熟悉项目的高级开发者 | 小白也能照样产出高质量提问 |

---

## 2. 触发条件

满足任一条即用 grill-deep-loop 替代 grill-deep：

- 阶段 2 步骤 4（逆向追问）：reverse-loop 产出后做最后一道验证；
- 阶段 3 步骤 2（人工 review 准备）：把 AI 当成"业务方代理"做模拟提问，捋出 review 时该问哪些问题；
- 接手陌生模块第一次做 grill；
- 业务方临时性 review 前的"自我演练"；
- 跑过一次 grill-deep 但问题质量明显偏弱。

不需要 Loop 的场景（用 grill-deep 即可）：

- 只想快速捋 3-5 个最关键问题；
- 已经做过一次系统 grill，只是补充某一个具体场景；
- 团队成员日常的快速对齐。

## 4. 输出

跑完一次 Loop，AI 必须输出 **3 段**：

### 4.1 按 6 维分组的问题清单

每条问题必须含 4 个字段：**问题 / 锚点 / 不答的后果 / 期望答案空间**。

### 4.2 提问质量自查表（核心新增产物）

```markdown
| 验收标准 | 评分 | 证据 / 卡点 |
|---|---|---|
| GQ1 边界条件维度 ≥2 问 | ✅ / ⚠️ / ❌ | 问题编号 |
| GQ2 异常路径维度 ≥2 问 | ✅ / ⚠️ / ❌ | ... |
| GQ3 事务与一致性 ≥2 问 | ✅ / ⚠️ / ❌ | ... |
| GQ4 幂等与重试 ≥2 问 | ✅ / ⚠️ / ❌ | ... |
| GQ5 并发与时序 ≥2 问 | ✅ / ⚠️ / ❌ | ... |
| GQ6 缓存 / 历史数据 ≥2 问 | ✅ / ⚠️ / ❌ | ... |
| GQ7 每问都有具体锚点 | ✅ / ⚠️ / ❌ | ... |
| GQ8 每问都有"不答的后果" | ✅ / ⚠️ / ❌ | ... |
```

### 4.3 优先级排序 + 需立即确认清单

```markdown
## 推荐 review 时优先问的 5 个问题
1. [GQ3] xxx（不答的后果：xxx）
2. [GQ4] xxx
...

## 业务方专属问题（技术方答不了的）
- [ ] ...
- [ ] ...
```

## 5. 通用 Prompt（核心模板，小白可直接复制粘贴）

```markdown
# 角色
你是一名熟悉 Java / SpringBoot + 业务领域的资深架构师，目标是用 Loop 方式对一个模块/功能做"杀伤力提问"。
你的工作不是"问得越多越好"，而是"问到不答就会出事的那几个真问题"。

# 任务
针对功能 / 模块 {名称}，生成一份高质量的 grill 问题清单，供人工 review 或业务方对齐使用。

# 流程（必须按此顺序，不要跳步）
1. 通读输入材料（reverse-loop / PRD / 代码），抽出本功能的核心场景
2. 按 GQ1–GQ6 六维度逐项生成问题，每维 ≥2 问
3. 为每条问题填写四字段：问题 / 锚点 / 不答的后果 / 期望答案空间
4. 按下方"验收标准"自查（覆盖度、锚点、后果是否到位）
5. 对 ⚠️/❌ 的维度补问（最多 2 轮）
6. 输出最终问题清单 + 自查表 + 优先级 5 问 + 业务方专属问题

# 6 个提问维度（必须每维 ≥2 问，否则记 ⚠️ 或 ❌）
- GQ1 边界条件：null / 空集合 / 最大值 / 最小值 / 越界 / 默认值
- GQ2 异常路径：超时 / 失败 / 第三方挂掉 / 输入畸形 / 鉴权失败
- GQ3 事务与一致性：事务边界 / 多步写入失败 / 跨库 / 分布式事务 / 数据漂移
- GQ4 幂等与重试：重复请求 / 重试策略 / 回调多次 / 消息重复 / 唯一键冲突
- GQ5 并发与时序：竞态 / 锁 / 状态机非法跳转 / 时序依赖
- GQ6 缓存 / 历史数据：缓存与库不一致 / 历史脏数据 / 字段语义演变 / 已废弃枚举值

# 提问质量硬性约束（每问都必须满足）
- 必须锚定到具体方法 / 字段 / 接口 / 状态枚举值，禁止"如何保证 XX"这种空泛问法
- 必须给出"如果不答会出什么事"（即使是模糊后果也要写）
- 必须给出"期望答案空间"（例如 "YES/NO" / "枚举其一" / "数值区间" / "需要业务方决策"）
- 禁止问已经写在 PRD / CLAUDE.md 中的内容（grill 是补不足，不是复读）

# 停止条件
- 全部 GQ1–GQ8 评为 ✅ → 立即停止
- 已自查修改 2 轮仍有 ⚠️/❌ → 停止，需人工确认清单中明确"哪个维度信息不足"
- 严禁为凑问题数量而提相似问题（同一问题换皮算 1 条）
```

## 8. 注意事项 / 常见陷阱

### 8.1 AI 做 grill 最容易掉的 5 个坑

| 坑 | 表现 | 应对 |
|---|---|---|
| **空泛问题** | "如何保证数据一致性" | GQ7 强制每问有具体锚点 |
| **维度偏科** | 全是边界问题，0 个幂等问题 | GQ1–GQ6 每维必须 ≥2 |
| **PRD 复读** | 问的内容 PRD 已经写了 | prompt 明确禁止 |
| **大杂烩** | 30 个问题不分优先级 | 强制输出"优先级 5 问" |
| **业务/技术不分** | 让开发回答只有业务方知道的问题 | 强制输出"业务方专属问题"清单 |

### 8.2 项目可加的自定义 GQ（按需追加）

- **GQ9 权限与多租户**：跨租户数据隔离、字段级权限、敏感数据脱敏；
- **GQ10 计费与对账**：金额精度、对账口径、月结日切换；
- **GQ11 国际化**：时区、币种、本地化字段长度；
- **GQ12 监控告警**：埋点是否到位、告警阈值、降级开关。

### 8.3 高价值问题的判断标准（人工抽查时用）

一个 grill 问题"值不值得问"，看四项：

1. **可回答性**：能用一句话或一个枚举值回答吗？（不能 = 太大）
2. **锚定具体**：能指到某行代码、某个字段吗？（不能 = 太虚）
3. **不答后果**：跳过此问，未来踩坑概率高吗？（低 = 不重要）
4. **业务/技术分清**：是问业务规则还是问技术实现？（混杂 = 浪费时间）

### 8.4 与其他 Skill 的衔接

- **跑完 grill-deep-loop 后** → 「优先级 5 问」直接进入人工 review 议程；
- **如果 grill 输出 ⚠️/❌ 太多** → 说明 reverse-loop 信息不足，回阶段 2 补；
- **业务方专属问题** → 在 review 会前发给业务方，避免临场发懵；
- **跟 to-prd 联动**：把 grill 答案直接合并到 PRD 的"待确认项"。

## 9. 一次最小可用示例

**输入**：

```
功能：订单 30 分钟未支付自动取消
材料：docs/modules/order.md（reverse-loop 产物）
代码：OrderTimeoutJob.java, OrderService.java, InventoryClient.java
schema：orders 表，状态字段 status int default 0
```

**期待 AI 输出（节选）**：

```markdown
## 一、按维度的问题清单

### GQ3 事务与一致性
| # | 问题 | 锚点 | 不答的后果 | 期望答案空间 |
|---|---|---|---|---|
| Q5 | OrderService.cancel() 内先 update status 再调 InventoryClient，外部调用失败时是否 rollback DB？ | OrderService.java:88 | 订单已取消但库存未释放 → 卖超 | YES/NO/补偿机制名称 |
| Q6 | InventoryClient.release() 失败是否抛出 IOException？还是返回 boolean？影响事务行为 | InventoryClient.java:42 | 业务层判断错误 → 错误的回滚行为 | "抛异常 / 返回 boolean / 其他" |

### GQ4 幂等与重试
| # | 问题 | 锚点 | 不答的后果 | 期望答案空间 |
|---|---|---|---|---|
| Q7 | OrderTimeoutJob 由两个实例同时触发（集群部署），如何防止同一订单被取消两次？ | OrderTimeoutJob.java:30 | 库存被释放两次 → 库存错乱 | "分布式锁名 / 状态机 CAS / 数据库唯一键" |
| Q8 | 取消失败重试 3 次仍失败时，订单进入什么状态？人工介入还是死信队列？ | OrderService.java:120 | 订单永远卡在中间态 | "状态枚举名 / 处理流程" |

## 二、自查表
| 维度 | 评分 | 证据 |
|---|---|---|
| GQ1 边界 | ✅ | Q1, Q2（null orderId、空订单列表）|
| GQ2 异常 | ✅ | Q3, Q4（超时、Client 返回 5xx）|
| GQ3 事务 | ✅ | Q5, Q6 |
| GQ4 幂等 | ✅ | Q7, Q8 |
| GQ5 并发 | ⚠️ | 仅 Q9 一条，缺"高峰期同一用户同时下单+取消"场景 |
| GQ6 缓存 | ✅ | Q10, Q11 |
| GQ7 锚点 | ✅ | 全部带文件:行号 |
| GQ8 不答后果 | ✅ | 全部填写 |

迭代轮次：2
是否达到停止条件：是（GQ5 已尽力补充但场景信息不足）

## 三、优先级 5 问
1. [GQ3 Q5] 取消失败时是否 rollback？（卖超风险）
2. [GQ4 Q7] 集群部署如何防重复取消？（库存错乱）
3. [GQ3 Q6] InventoryClient 失败的返回形式？（事务回滚不一致）
4. [GQ4 Q8] 重试 3 次仍失败的兜底？（订单卡死）
5. [GQ6 Q10] 已取消订单的缓存清理时机？（前端显示不一致）

## 四、业务方专属问题
- [ ] 30 分钟超时窗口是否对所有商品类型一致？（生鲜可能 5 分钟）
- [ ] 已支付但未发货的订单是否纳入取消逻辑？
- [ ] 取消订单后是否自动通知用户？通知渠道？
```

这就是一次合格的 grill-deep-loop 输出 —— **每条问题都有锚、有后果、能回答**。
